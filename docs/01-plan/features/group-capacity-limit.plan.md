# 그룹별 차수 정원 제한 — Plan

---

## Executive Summary

| 관점 | 내용 |
|------|------|
| **Problem** | 차수에 그룹을 배정할 수 있으나 그룹당 할당 인원 한도가 없어, 특정 부서가 정원을 독점하는 상황을 방지할 수 없음 |
| **Solution** | 차수의 그룹 배정 UI에 그룹별 할당 한도(limit) 입력 필드를 추가하고, 한도 합산이 총 정원을 초과하지 않도록 유효성 검증 |
| **기능/UX 효과** | 관리자가 "개발팀 10석 / 디자인팀 8석" 식으로 차수별 부서 쿼터를 직접 지정 가능. 사용자 신청 시 그룹 초과 여부 즉시 피드백 |
| **Core Value** | 부서별 교육 기회 균등 배분 — 기업 환경에서 필수적인 그룹 쿼터 관리를 프로토타입 단계에서 설계 확정 |

| 항목 | 내용 |
|------|------|
| **WHY** | 데이터 모델 미확정 시 Phase 2에서 전면 리팩토링 필요. 지금 결정이 전체 수강 신청 로직에 영향 |

| **SCOPE** | admin/offline-builder.html (그룹 배정 UI + 유효성), user/course-detail.html (그룹 초과 안내 — 별도 Phase) |

## 1. 요구사항

### 핵심 요구사항

| # | 항목 | 설명 | 필수 여부 |
|---|------|------|:--------:|
| R-01 | 그룹별 할당 한도 입력 | 차수의 그룹 배정 영역에서 각 그룹에 숫자 입력 필드 추가 | 필수 |
| R-02 | 한도 합산 유효성 | 모든 그룹 limit 합산 ≤ option.capacity. 초과 시 저장 불가 + 에러 표시 | 필수 |
| R-03 | 총 정원 자동 연동 | 그룹이 배정된 경우, 각 limit 입력 시 총 정원 표시 업데이트 (시각적 피드백) | 필수 |
| R-04 | 미배정 잔여석 표시 | 총 정원 - 할당 합산 = 미배정 잔여석으로 별도 표시 | 권장 |
| R-05 | 그룹 없을 때 기존 동작 유지 | 그룹 미배정 차수는 기존 capacity 단일 정원 방식 그대로 | 필수 |

### 데이터 모델 변경

**현재 구조**:
```js
option.groups = [
  { id: 'g1', name: '개발팀', count: 12 },  // count = 그룹 전체 인원수
]
```

**변경 후 구조**:
```js
option.groups = [
  { id: 'g1', name: '개발팀', count: 12, limit: 10 },  // limit = 이 차수에서 이 그룹에 배정된 좌석 수
]
```

- `limit`: 차수 내 해당 그룹에 배정할 최대 좌석 수 (0 = 미설정, 제한 없음)
- `count`: 그룹 전체 인원 수 (기존 필드, 변경 없음)

### 비기능 요구사항

| 항목 | 기준 |
|------|------|
| 입력 유형 | 양의 정수. 빈 값 = 0으로 처리 |
| 유효성 타이밍 | 입력 즉시(onChange) + 저장 버튼 클릭 시 |
| 기존 데이터 호환 | limit 필드 없는 기존 그룹 객체 → limit: 0으로 취급 |

## 4. UI 상세 명세

### 그룹 배정 칩 (강제 할당 모드, 커리큘럼 사이드바)

**현재**: 그룹명 칩 + 제거(×) 버튼

**변경 후**:
```
[ 개발팀  [10] ×]   ← 숫자 인풋 (limit 입력)
[ 디자인팀 [ 8] ×]
[ + 그룹 추가 ]

총 정원 30석 | 할당 18석 | 미배정 12석
```

- limit 입력은 칩 내부 작은 number input (`w-12`, `text-center`)
- 미배정 잔여석 = capacity - Σlimit. 음수가 되면 빨간색 경고
- 0 입력 = "한도 없음"으로 처리 (해당 그룹은 잔여석 내에서 자유 신청)

### 유효성 에러 메시지

```
그룹 할당 합산(38석)이 총 정원(30석)을 초과합니다.
```

## 5. 리스크

| 리스크 | 가능성 | 대응 |
|--------|:------:|------|
| 기존 groups 배열 변경으로 handleAddGroup/handleRemoveGroup 로직 영향 | 중 | limit 필드는 선택적으로 추가. 미존재 시 0으로 fallback |
| capacity 자동 계산 로직과 충돌 | 중 | 그룹 배정 시 capacity 자동 계산은 유지, limit 별도 필드로 분리 |
| limit=0 의미 모호 | 저 | 0 = 제한 없음으로 명확히 정의. UI에 "—" 표시 |

---

## 6. 성공 기준

1. 그룹 칩에서 limit 숫자를 입력할 수 있다
2. limit 합산 > capacity 시 저장 버튼이 비활성화되거나 에러 메시지가 표시된다
3. 미배정 잔여석이 실시간으로 업데이트된다
4. 그룹이 없는 차수는 기존 동작과 동일하게 작동한다
5. 기존 groups 배열(limit 없는 구조)을 그대로 사용해도 오류 없음

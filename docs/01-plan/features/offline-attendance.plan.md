# 오프라인 코스 출석·수료 관리 — 공식 기획 Plan

---

## Executive Summary

| 관점 | 내용 |
|------|------|
| **Problem** | 오프라인 교육 현장에서 출석 확인이 수기 명단 체크로 이뤄져 집계 오류·누락 발생. 회차 마감 후 수료 처리가 HRD 담당자의 수동 작업에 의존 |
| **Solution** | QR 스캔으로 출석 자동 체크, 5종 상태 수동 수정, 회차 마감 시 미확인→결석 일괄 전환. 수료 처리는 Phase 2로 이월 |
| **기능/UX 효과** | 강사가 현장에서 QR 라이브 모드를 띄우면 학습자가 앱으로 스캔 → 실시간 출석 집계. HRD 담당자는 관리 탭에서 수동 보정 후 회차 마감 |
| **Core Value** | "현장에서 엑셀 없이 출석을 처리한다" — offline-builder와 함께 Phase 1 완결. 수료 확정 기능은 Phase 2에서 추가 |

## 1. 사용자 인텐트 발견

### 핵심 문제
오프라인 교육의 출석 확인·집계·수료 처리가 수기 방식으로 운영되어 오류·누락이 발생하고 HRD 담당자의 수작업 부담이 큼.

### 주요 사용자

| 사용자 | 역할 | 핵심 니즈 |
|--------|------|-----------|
| **HRD 담당자** | 출석 수정, 회차 마감, 수료 처리, 통계 확인 | 수동 개입 최소화, 한눈에 집계 |
| **강사** | 현장 QR 화면 운영, 실시간 출석 확인 | 빠른 QR 진입, 실시간 현황 |
| **상위 관리자** | 차수별 출석률·수료율 리포트 | 집계 자동화 |

### 성공 기준
- 강사가 라이브 모드로 QR을 띄우면 학습자 스캔 즉시 출석 반영
- HRD 담당자가 회차 마감 → 미확인 일괄 결석 전환까지 화면에서 완료
- 차수별 출석률이 자동 집계되어 별도 엑셀 불필요
- 수료 처리(확정 버튼)는 Phase 2로 이월 — 사이드이펙트 없음 확인 (2026-04-28)

## 3. 아키텍처 결정사항

### 데이터 구조

```
COURSES_DB[courseId]
 └── packages[] (차수)
       ├── sessions[] (회차)
       └── studentMap: { pkgId: [userId, ...] }

attendance: { [sessId]: { [userId]: status } }
status: 'pending' | 'attended' | 'late' | 'early_leave' | 'absent'
```

### 탭 구조

| 탭 | ID | 주요 기능 |
|----|----|-----------|
| 요약 | summary | 코스 현황, 알림, D-day |
| 수강 | students | 수강생 목록, 차수별 필터 |
| 출석 | attendance | 회차별 출석 체크, 라이브 QR |
| 설문 | survey | (Phase 2) |

### 기술 스택
- React 18 (Babel standalone, ESM CDN)
- Tailwind CSS (CDN)
- Lucide React 0.292.0
- BroadcastChannel API (데모 시나리오 제어)
- 단일 HTML 파일

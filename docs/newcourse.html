NEWCOURSE · 기획설계서

newcourse

newcourse는 터치클래스의 새로운 코스 시스템입니다. 영상·문서·시험·설문·오프라인 출석을 동일 슬롯 모델 위에서 자연스럽게 표현하며, 기존(v1 페이지 기반) 코스와 별도 시스템으로 영구 공존합니다.

00 / Summary한눈에 보기

한 줄 룰

터치클래스는 레거시와 newcourse를 동등한 옵션으로 운영한다. newcourse는 "에디터 없는 깔끔한 운영" + "새로운 코스 유형(오프라인·블렌디드·단독 에셋)이 비로소 가능"한 그린필드 시스템이며, 두 시스템 간 코스·데이터 공존은 없다.

이 결정의 이유 다섯 가지

레거시는 비즈니스 주력 — 흔들지 않는다.

매출 비중은 레거시에 있다. 신규 모델을 위해 레거시를 수정·확장하면 리스크가 너무 크다. 레거시는 유지 운영, 신규 기능은 newcourse에만 추가한다.

newcourse의 진짜 가치 = 새 코스 유형이 비로소 가능해진다.

"에디터 걷어내기"는 수단이고, 본질은 오프라인·블렌디드·단독 에셋 코스가 동일 모델 위에서 자연스럽게 표현된다는 점이다. 레거시 페이지 모델로는 못 만들던 코스 유형들이다.

"기능 적음 + 단단함" — 보수적 고객 타깃 가설.

기업교육 운영자 다수가 "영상만 올리고 끝내고 싶다"고 한다. 복잡한 저작도구는 학습 효과보다 운영 부담을 키운다. 보수적·단순 운영 선호 고객사에 newcourse의 단순성이 더 매력적이라는 가설.

두 시스템 코스 공존 비용 > 분리 비용.

한 코스 안에 레거시 데이터와 newcourse 데이터를 섞으면 진도율 정합성 사고와 CS 부담이 폭증한다. 시스템 수준에서는 두 제품을 동시 운영하되, 코스/데이터 수준에서는 완전 분리한다.

그린필드 = 진도 정합성·운영 모델·UX 자유.

레거시 호환 책임이 없으니 슬롯 단위 진도율·결정적 정합성·등록 채널 3종·개인별 수강 기간·만료 정책을 처음부터 깔끔히 박을 수 있다.

기획·영업·운영 누구나 이 글만 읽으면 newcourse를 왜 만드는지·어떻게 가는지 이해합니다. 모든 표현은 확정된 방향입니다.

1. 지금 우리가 겪는 문제

지금 터치클래스는 옛 저작도구 중심으로 돌아갑니다. 영상 하나 올리는 데도 저작도구에 들어가 페이지를 만들고 그 결과물을 코스에 붙입니다. 다음 문제들이 커지고 있습니다.

  • 단순 영상 하나 올리기에도 절차가 복잡합니다
  • 학습자 화면이 콘텐츠 종류와 맞지 않아 불편합니다
  • 새 콘텐츠 종류를 추가할 때마다 개발 부담이 커집니다
  • 오프라인 출석 같은 요소를 코스에 자연스럽게 섞기 어렵습니다
  • 운영 중 목차가 바뀌면 진도율·통계가 꼬입니다

문제의 본질은 "기능 부족"이 아닙니다. 구조가 너무 옛 방식에 맞춰져 있어 새 요구를 받기 어렵습니다.

2. 우리가 원하는 방향

새 시스템을 통째로 만들지 않습니다. 반대입니다. 기존 시스템을 그대로 유지하면서, 사용자는 더 쉽게 쓰고, 운영은 더 유연하게 만듭니다.

  • 레거시 콘텐츠는 계속 운영합니다
  • 영상·오디오·이미지·문서 같은 단일 파일은 쉽게 올립니다
  • 오프라인 강의·출석도 같은 코스 안에 자연스럽게 섞습니다
  • 학습자에겐 콘텐츠 종류에 맞는 편한 화면을 보여줍니다
  • 기존 통계·수료 구조는 그대로 유지합니다
  • 개발 비용은 최소화합니다

3. 가장 좋은 해결 방향

기존 레거시를 그대로 두고, 그 위에 더 넓은 운영 틀을 얹습니다. 옛 상자를 없애지 않습니다. 영상·문서·오프라인 기록까지 같이 담는 더 큰 선반 체계를 만듭니다. 그 안에서 레거시는 한 종류로 남고, 영상도 하나의 종류, 문서도 하나의 종류, 오프라인 출석도 하나의 종류로 함께 관리합니다.

4. 꼭 짚어둘 점 — 옛 콘텐츠를 새 구조로 자동 옮기는 작업은 당장 안 합니다

"새 구조로 가면 기존 레거시 콘텐츠도 자동으로 옮겨지나요?" 가장 많이 헷갈리는 부분입니다. 답은 당장 안 합니다.

  • 레거시(페이지 방식)와 새 구조(학습 항목 방식)는 안에 담긴 내용의 모양이 다릅니다. (예시: 데이터 모델 차이 — 자세한 사항은 개발팀에서 결정합니다)
  • 이미 운영 중인 레거시 코스에 강제로 적용하면 학습 기록·진도율·수료가 깨집니다
  • 그래서 레거시 콘텐츠는 그 자리에 그대로 둡니다. 새 구조는 새로 만드는 코스에서만 적용합니다
  • 운영자가 새 코스를 만들 때 "레거시 방식으로 만들지" 또는 "새 방식으로 만들지"를 선택합니다. 둘 다 똑같이 사용 가능합니다
  • 레거시 → 새 구조 자동 옮기기 기능 (예시: 마이그레이션 기능 — 자세한 사항은 개발팀에서 결정합니다) 은 현재 계획에 없습니다

5. 앞으로 시스템을 어떻게 이해해야 하는가

앞으로 콘텐츠를 "저작도구 결과물"로만 보지 않습니다. 코스 안에 들어가는 모든 학습 요소를 하나의 공통 개념 — "학습 항목" 으로 봅니다.

  • 기존 레거시 콘텐츠
  • 단일 동영상 · 단일 오디오 · 이미지 자료 · 문서형 콘텐츠
  • 마크다운으로 만든 새 콘텐츠
  • 오프라인 강의 안내 · 오프라인 출석 체크
  • 퀴즈 · 평가

앞으로 코스는 "페이지형 콘텐츠 묶음"이 아닙니다. 여러 종류의 학습 항목을 순서대로 담아 운영하는 구조입니다.

6. 왜 이 방식이 가장 좋은가

기존 것을 거의 안 건드립니다

레거시는 잘 돌아갑니다. 통계·학습 기록이 많이 쌓여 있습니다. 크게 바꾸면 위험·비용이 큽니다. 그래서 레거시도 "콘텐츠 종류 중 하나"로 포함합니다.

관리자 사용이 쉬워집니다

영상 교육은 영상 업로드 → 제목 → 학습 시간·수료 조건 → 코스에 넣기로 끝납니다. 저작도구에 들어가 페이지를 만드는 단계가 사라집니다.

학습자 화면이 편해집니다

콘텐츠마다 가장 잘 맞는 화면을 보여줍니다. 동영상은 동영상 화면, 문서는 문서 화면, 오프라인 출석은 출석 체크 화면입니다.

오프라인 출석도 코스 안에 넣습니다

사전 안내문 → 오프라인 강의 참석 → 출석 체크 → 사전 영상 → 현장 실습 자료 → 사후 테스트, 이렇게 한 코스로 묶습니다.

기존 통계를 유지합니다

통계를 새로 만들지 않습니다. 새 콘텐츠들의 학습 기록을 기존 체계에 맞는 신호로 흘려보냅니다. 영상 시청은 "학습 시작·진행·완료", 문서 열람은 "열람·체류·완료", 오프라인 출석은 "완료 처리"로 흘려보냅니다. (예시: 학습 이벤트 신호 매핑 — 자세한 방식은 개발팀에서 결정합니다) 겉모습은 바뀌어도 안쪽 운영 숫자는 끊기지 않습니다.

7. 가장 중요한 운영 원칙 7가지

원칙 1레거시는 유지합니다 — 기존 콘텐츠를 계속 운영합니다
원칙 2신규는 쉽게 등록합니다 — 영상·문서·오디오를 빠르게 올립니다
원칙 3코스는 여러 종류의 학습 항목을 담습니다 — 레거시·신규·오프라인 출석을 한 코스에 섞습니다
원칙 4학습자 화면은 콘텐츠 종류에 맞게 분기합니다 — 하나의 화면으로 다 해결하지 않습니다
원칙 5통계는 기존 기준을 유지합니다 — 새 콘텐츠도 운영 숫자가 끊기지 않게 이어갑니다
원칙 6운영 중 코스 변경은 신중하게 다룹니다 — 이미 학습 중인 사람의 진도·수료가 깨지지 않게 코스 버전 관리로 다룹니다 (예시: 코스 버전 관리 — 자세한 방식은 개발팀에서 결정합니다)
원칙 7레거시 → 새 구조 자동 옮기기는 당장 안 합니다 — 안에 담긴 내용의 모양이 달라 자동 변환이 안 됩니다 (예시: 데이터 모델 차이 — 자세한 사항은 개발팀에서 결정합니다). 레거시는 그 자리에 그대로 두고, 새 구조는 신규 코스에서만 적용합니다

8. 코스 변경 시 왜 조심해야 하는가

운영 중인 코스 목차를 바꾸거나 항목을 지우면 이미 그 항목을 들은 사람의 기록·진도율·수료 기준이 흔들립니다. 그래서 운영 중 코스는 코스 버전 관리로 다룹니다. (예시: 코스 버전 관리 — 자세한 방식은 개발팀에서 결정합니다)

  • 이미 수강 중인 사람은 기존 코스 기준으로 계속 갑니다
  • 새로 들어오는 사람만 바뀐 코스를 적용받습니다

9. 관리자에게 보이는 모습

복잡한 저작 시스템이 아닙니다. 훨씬 단순한 "코스 구성 화면"입니다. 동영상 추가 → 문서 추가 → 레거시 콘텐츠 추가 → 오프라인 출석 추가 → 퀴즈 추가, 이걸 순서대로 배치합니다. 콘텐츠를 복잡하게 만드는 화면이 아닙니다. 필요한 학습 요소를 골라 코스를 구성하는 화면입니다.

10. 학습자에게 보이는 모습

안내 보기 → 영상 시청 → 문서 읽기 → 오프라인 참석 → 출석 체크 → 복습 콘텐츠 → 테스트 응시. 이 모두가 따로 놀지 않습니다. 하나의 교육 여정처럼 보입니다. 학습 경험도 좋아지고 운영도 쉬워집니다.

11. 한 문장 요약

터치클래스는 기존 레거시를 없애지 않습니다. 유지한 채, 영상·문서·오프라인 출석까지 함께 담는 '통합 학습 항목 구조' 로 코스 운영을 넓힙니다.

12. 최종 정리

새 저작도구 하나를 만들지 않습니다. 코스를 운영하는 기준 자체를 넓힙니다. 과거는 "레거시 콘텐츠를 코스에 넣는 시스템"입니다. 앞으로는 "여러 종류의 학습 요소를 하나의 코스로 운영하는 시스템" 입니다.

이 전환은 다음 조건을 모두 만족합니다.

  • 기존 레거시 유지
  • 개발 비용 최소화
  • 관리자 사용 단순화
  • 학습자 화면 개선
  • 온라인·오프라인 혼합 학습 지원
  • 기존 통계·수료 구조 유지

레거시를 유지하면서 그 위에 새로운 통합 운영 구조를 얹습니다. 이것이 확정된 방향입니다.

01 / Valuenewcourse가 비로소 가능하게 하는 것

레거시 페이지 모델로는 못 만들던 코스 유형들. 별도 모델이 아닌 동일 슬롯 시스템 위에서 표현됩니다.

코스 유형 3종

TYPE 1
콘텐츠 코스

영상·문서·시험·설문 슬롯으로 구성. 가장 기본 유형. 슬롯 1개짜리 단독 에셋 코스(영상 1개 · 시험 1개 · 설문 1개)도 이 유형으로 표현됩니다.

슬롯 예시: video × N · page × N · quiz × 1 · survey × 1
TYPE 2
오프라인 코스

출석 슬롯만으로 구성된 코스. 강사·교실·날짜 기반 운영. 레거시에서는 별도 시스템으로 운영하던 영역이 newcourse 안에 자연스럽게 들어옵니다.

슬롯 예시: offline_attendance × N
TYPE 3
블렌디드 코스

온라인 + 오프라인 혼합. 영상 사전 학습 → 오프라인 워크숍 → 시험 같은 흐름이 하나의 코스 안에서 표현됩니다.

슬롯 예시: video → offline_attendance → quiz
핵심. 코스 유형별 코드 분기 없음. 진도율·수료·통계 로직은 단 1벌. 유형은 "어떤 슬롯들로 구성됐는가"의 결과일 뿐이며, 시스템 내부에서는 별도 모델로 취급하지 않습니다.

★ / Terminology용어 — 운영팀 언어 vs 개발 언어

같은 객체를 운영팀과 개발팀이 다르게 부릅니다. 본 문서는 운영팀 언어를 우선으로 쓰고, 필요할 때만 괄호로 개발 용어를 병기합니다.

운영팀 · 학습자 언어
코스 → 커리큘럼 → 에셋
관리자 화면 · 학습자 화면 · 안내 메일에서 사용
코스 ├─ 메타·정책 │ ├─ 이름·설명·강사 │ ├─ 최대 운영 기간 │ └─ 코스 전체 수료 기준 ├─ 커리큘럼 (학습 순서·구성) │ └─ 에셋 + 수료 조건 │ ├─ 영상 (시청률 90%) │ ├─ 시험 (60점 통과) │ ├─ 설문 (제출 완료) │ ├─ 문서 (조회 완료) │ ├─ 출석 (출석률 100%) │ └─ 외부 링크 (방문 기록) └─ 수강 등록 정책 ├─ 채널 A: 관리자 수동 등록 (명단·그룹) ├─ 채널 B: 학습자 자율 신청 (신청 가능 그룹) ├─ 채널 C: 가입 시 자동 등록 (그룹 매핑) └─ 수강 기간 (날짜 / 등록일+N일 / 무제한)
개발 · 데이터 모델 언어
Course → Curriculum → Slot
코드 · API · DB 스키마에서 사용
Course ├─ meta · policy │ ├─ title · description · instructor │ ├─ maxRunDays │ └─ courseCompletionRule ├─ Curriculum (independent object) │ └─ Slot[] (sequence) │ ├─ contentRef → Asset │ │ (video/quiz/survey/page/ │ │ offline_attendance/external_link) │ └─ completion policy │ (watchRate / passScore / ...) └─ EnrollmentPolicy ├─ adminAssign (manual: roster / group) ├─ selfSignup (allowedGroups[]) ├─ autoOnSignup (groupMapping[]) └─ studyWindow (fixed / days / unlimited) Enrollment[] (per learner) ├─ userId · courseId ├─ channel (A / B / C) ├─ enrolledAt └─ individual studyWindow

1:1 매핑

운영팀 언어 개발 언어 · 설명
코스 Course · 메타·정책 컨테이너. 학습자도 "코스"로 인식
커리큘럼 Curriculum · 학습 순서·구성 단위. 독립 객체
에셋 Slot.contentRef → Asset · 실제 학습 콘텐츠 (영상·시험·설문·문서·출석·외부 링크)
수료 조건 Slot.completion · 각 에셋의 완료 기준 (시청률·통과 점수·출석률 등)
(통합 개념 없음) Slot · 에셋 + 수료 조건 + 커리큘럼 안 순서 위치를 묶은 1개 단위
수강 등록 정책 EnrollmentPolicy · 코스에 학습자가 어떻게 들어오는지 정의. 채널 3종(관리자 수동·자율 신청·가입 시 자동) + 수강 기간
등록(수강생) Enrollment · 학습자 ↔ 코스 1:1 연결 기록. 각자 등록일·개인 수강 기간을 가짐
그룹 Group · 조직 단위(영업1팀·신입사원 등). 수강 권한·자동 등록 매핑·일괄 강제 등록의 기준
핵심. 운영자가 "영상 에셋을 커리큘럼에 추가하면서 시청률 90% 수료 기준을 설정한다"는 하나의 동작이, 개발 입장에서는 Slot 객체 1개 생성으로 매핑됩니다. 즉 "에셋 + 수료 조건 = 슬롯 1개"입니다.

차수(회차) 개념은 newcourse Phase 1에 포함되지 않습니다. 기업 LMS는 학습자 변동·정책 변화가 잦아 동기 학습 차수보다 개인별 등록·개인별 수강 기간이 정론입니다. 정기 갱신·동기 학습이 정식 요구로 올라오면 그때 옵션 객체로 추가합니다.

02 / Structure시스템 구조 — 코스 · 커리큘럼 · 수강 등록 정책

코스는 메타·커리큘럼·수강 등록 정책 3블록으로 구성됩니다. 기업 LMS의 변동성에 맞춰 차수 객체는 두지 않습니다.

코스 (Course)
메타·정책 컨테이너
  • 메타: 제목·설명·분류·강사
  • 정책: 최대 운영 기간 · 코스 전체 수료 기준
  • 참조: 커리큘럼 1개
  • 상태: 초안 / 운영중 / 아카이브
커리큘럼 (Curriculum)
학습 순서·구성
  • 에셋 + 수료 조건 묶음 N개 (순서대로)
  • 버전: draft / published / deprecated
  • 발행 시 동결 (편집 차단 · 진도율 보존)
  • 새 버전 발행 시 기존 등록자는 발행 시점 버전 유지
수강 등록 정책 (EnrollmentPolicy)
학습자가 어떻게 들어오나
  • 채널 A: 관리자 수동 등록 (명단·그룹)
  • 채널 B: 학습자 자율 신청 (신청 가능 그룹)
  • 채널 C: 가입 시 자동 등록 (그룹 매핑)
  • 수강 기간: 날짜 / 등록일+N일 / 무제한

왜 이 구조인가

  • 레거시 결함 — 코스↔커리큘럼 1:1 임베드: 커리큘럼이 코스의 종속 자식이라 재사용 불가. 같은 과정을 매년 새 코스로 복제 → 데이터 파편화.
  • 레거시 결함 — 페이지 단위 진도율: 분모 가변. 콘텐츠 편집이 진도율을 흔든다.
  • newcourse 해결: 커리큘럼 독립 객체 + 다대다 참조Phase 3, 슬롯 단위 진도율, 발행 버전 동결로 정합성 보존.
  • 차수 객체 의도적 제외: 기업 LMS는 학습자 변동·정책 변화가 잦아 동기 학습 차수보다 개인별 등록·개인별 수강 기간이 정론. 정기 갱신·동기 학습이 정식 요구로 올라오면 Phase 2+ 옵션으로 추가.

03 / Asset에셋과 수료 조건 (개발: Slot)

에셋 = 학습 콘텐츠. 수료 조건 = 그 에셋을 "완료했다"고 판정할 기준. 둘이 묶여 1개 슬롯이 됩니다. 진도율은 이 슬롯 단위로 셉니다.

에셋 6종 + 각 수료 조건

에셋 종류수료 조건 · 비고
영상 (video)시청률 ≥ 90% · PDF는 별도 에셋 대신 문서 에셋의 sub-format으로 흡수 예정검토중
문서 (page)조회 완료 · 텍스트·이미지·임베드. 노션 스타일 본문 작성
시험 (quiz)통과 점수 ≥ 60 · 시험·퀴즈
설문 (survey)제출 완료 · 설문
출석 (offline_attendance)출석률 = 100% · 오프라인 출석
외부 링크 (external_link)방문 기록 · 외부 링크 (강제 조건 없음)
레거시와의 결정적 차이. 레거시는 페이지 단위 진도율이라 분모가 가변이다. 콘텐츠 편집이 진도율을 흔든다. newcourse는 슬롯 단위(에셋 + 수료 조건 묶음)로 카운트하므로 에셋 내부 콘텐츠(영상 교체·문서 수정 등)는 자유롭게 편집해도 진도율이 흔들리지 않는다.

슬롯 부가 정책 — 진입·공개 제어

  • 선수조건 (prerequisiteIds) — 앞 슬롯을 완료해야 해당 슬롯 진입 가능
  • 드립 학습 (releaseAfter) — 등록일 또는 학습시작일 기준 N일 후 공개
  • 드립 기준일 (dripLearning.baseDate) — 등록일 vs 학습시작일 선택
  • 순차 학습 잠금 (pageSequenceLock) — "이전 슬롯 완료 후 다음 진입" 의미 명확화 필요운영팀 협의

04 / Operations운영 정책

코스 만료·수강 등록 채널·수강 기간·컴플라이언스. 진도율 정합성을 깨지 않으면서 운영 자유도를 확보합니다.

코스 최대 운영 기간 (maxRunDays)

기본값365일 권장
의미이 기간 후 코스 자동 아카이브. 신규 등록 중단
역할데이터 라이프사이클·폐기 정책의 앵커
프리셋90 / 180 / 365 / 730일

수강 등록 채널 3종

현재 운영 케이스 4가지(관리자 일괄 강제·관리자 그룹별 강제·자율 신청·가입 시 자동)를 다음 3개 채널로 통합합니다.

A · 관리자 수동 등록관리자가 명단(엑셀 업로드) 또는 그룹을 선택해 일괄 등록. 100~1000명 단위 강제 등록 케이스 포함
B · 학습자 자율 신청코스를 "신청 가능"으로 열어둠. 신청 권한을 특정 그룹으로 제한 가능. 학습자가 임의 시점에 신청 → 등록
C · 가입 시 자동 등록회원가입 시 부여되는 그룹과 코스를 매핑. 신규 가입자가 그룹에 속하면 자동으로 해당 코스에 등록. 예: "신입사원" 그룹 가입 → "신입 온보딩" 자동 등록

※ 채널 A·B·C는 동시 활성 가능. 한 코스에 여러 채널을 함께 열어둘 수 있음.

수강 기간 (등록 단위)

fixed코스에 지정된 시작·종료 일자 동안 학습 가능. 모든 등록자가 같은 윈도우 공유
days각 학습자 등록일 기준 N일. 등록 시점이 다르면 만료 시점도 다름
unlimited코스 자체 만료(maxRunDays) 전까지 무제한

컴플라이언스

  • 시간대 제한: 특정 요일·시간만 학습 가능 (법정 의무교육 대응)
  • 유휴 모니터링: 일정 시간 무동작 시 학습 시간 카운트 중단
  • 서약서: 학습 시작 전 동의서 표시 및 동의 기록
참고. 시간대 제한·유휴 모니터링·서약서는 법정 의무교육(성희롱·개인정보·산안 등) 운영 필수 옵션. 일반 코스에는 기본 비활성.

차수가 없다는 것의 의미. 같은 의무교육을 매년 돌리고 싶을 때, 레거시는 매년 새 코스를 복제했지만 newcourse는 동일 코스를 유지하고 등록자만 갱신합니다. 1년차 등록자는 만료 시 자동 등록 해제, 새 회원은 채널 C로 자동 재등록되는 식. 정기 갱신 추적이 정식 요구가 되면 그때 옵션 객체로 추가합니다.

05 / Workspaces4영역 분리

07 / RoadmapPhase 로드맵

단계별 적용 범위. Phase 1 본문, Phase 2 이후는 적용 시점 미확정.

PHASE 1
기본 운영
에셋 + 수료 조건 시스템 + 등록 채널 3종

코스 유형 3종 · 에셋 6종 + 수료 조건 · 슬롯 단위 진도율 · 수강 등록 정책(채널 A/B/C) · 최대 운영 기간 · 기본 빌더 3종.

  • 코스 / 커리큘럼 / 수강 등록 정책 3블록 (개발: Course / Curriculum / EnrollmentPolicy)
  • 슬롯 단위 진도율 · 결정적 정합성
  • 등록 채널 3종 (관리자 수동 / 자율 신청 / 가입 시 자동) + 그룹 매커니즘
  • 개인별 수강 기간 (등록일 기준)
  • 온라인 / 오프라인 / 블렌디드 빌더 분리 진입
PHASE 2
확장 정책 시점 미정
선수조건 · 드립 학습 · 컴플라이언스

슬롯 진입 조건과 시간대 정책. 법정 의무교육 운영 요구 수용.

  • 선수조건 — 단순 앞 슬롯 1:1 (개발: prerequisiteIds)
  • 드립 학습 — 등록일·학습시작일 기준 (개발: releaseAfter / dripLearning)
  • 시간대 제한 · 유휴 모니터링 · 서약서
  • 커리큘럼 버전 발행 로직 (draft → published → deprecated)
PHASE 3
운영 자유도 시점 미정
학습자별 예외 + 커리큘럼 다대다 + (옵션) 차수

학습자별 면제·대체·예외 동결. 커리큘럼 다대다 참조 허용. 차수(회차)는 정식 요구 시 옵션 객체로 추가.

  • 학습자별 예외 처리 (개발: EnrollmentPlan) — 면제·대체
  • 커리큘럼 독립 + 다대다 참조
  • 차수(회차) 옵션 객체정식 요구 시 — 동기 학습·회차별 정원·강사 배정·정기 갱신이 필요한 케이스에 한해
PHASE 4
정합성 인프라 시점 미정
학습 증거 저장소 · 정합성 분리 · 학습 경로 그래프

학습 증거 영속화, 정합성 처리 분리, 분기·대체 경로 등 복잡한 학습 경로.

  • 학습 증거 저장소 (개발: Evidence Store) — 모든 학습 이벤트 영속
  • 정합성 계산 분리 (개발: Fulfillment Layer)
  • 학습 경로 그래프 (개발: Learning Graph) — 분기·대체·재시도

미해결 논의 — 다음 회의 안건

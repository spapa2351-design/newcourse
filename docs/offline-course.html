3. 서비스 구조

시스템이 어떻게 구성되는가. 개발팀이 반드시 알아야 할 계층 구조.

3.1 핵심 계층 구조
핵심 개념: 코스→커리큘럼→슬롯→차수→회차 계층을 이해해야 모든 기능이 보입니다.
코스 (Course) 무엇을 배우나 — 제목·강사·분류·신청 설정
└── 커리큘럼 (Curriculum) 어떻게 배우나 — 슬롯 순서/구성
└── 슬롯 (Slot) 개별 학습 단계 (타입: 오프라인출석 / 영상 / 시험 / 설문)
└── 차수 (Session) 운영 회기 — 날짜·정원·강사. Phase 1: 오프라인 슬롯만
└── 회차 (Class) 개별 수업일 — 날짜·시간·장소. 여기에 출석 기록

4.3 회차 종료 (시간 기반 자동)

"마감"이라는 별도 이벤트와 버튼이 없습니다. 회차는 수업 종료 시각이 지나면 자동으로 종료되며, 운영자가 별도로 닫는 행위는 없습니다. 미확인 인원은 운영자가 직접 정리합니다.

운영 가이드: "미확인" 인원은 운영자가 직접 정리(출석/지각/결석)합니다. Phase 1엔 자동 결석 처리 없음 — 운영자 수동 처리만.
4.3.1 회차 종료, 쉽게 풀어보기

학습자가 자주 묻는 "오늘 못 가면 한 회차 까이는 거예요?" — "마감"이라는 별도 사건은 없어요. 수업 시간이 끝나면 그 회차도 자동으로 끝나는 것으로 통일됩니다.

🎯 한 줄 정의
회차 종료 = 수업 종료 시각이 지난 시점.
별도의 "마감 버튼"이나 운영자 행동 없이 자동으로 종료됩니다. 그 시점부터 출석률 계산에 그 회차가 들어가고, 미확인 인원은 운영자가 직접 정리합니다.

📊 운영자 화면에서 일어나는 일

📢 학습자 알림 — 즉시성만 Phase 1, 나머지 Phase 2
✅ Phase 1 — 즉시성 알림
  • 출석 처리됨 (QR 찍자마자) — "출석 처리됐어요"
  • 출석률 표시 (수업 종료 즉시) — 학습자 화면 갱신
⏸ Phase 2 이월
  • 미확인 처리 중 알림 (약한 알림)
  • 결석 확정 알림
  • HR 시스템 연동 (외부 시스템 통보)
✏️ 종료 후에도 출석은 고칠 수 있어요
• 결석 → 출석으로 바꾸는 건 종료 후에도 가능 (관리자만)
• 바꾸는 즉시 출석률 다시 계산
• "마감 자체를 풀기"는 개념 자체가 없어요 — 마감 사건이 없으니까. 수업 종료 시각이 지나면 자동 종료, 그 사실은 변하지 않음
• Phase 2 수료 확정 후 정정해도 수료 상태는 자동으로 안 바뀝니다 (별도 처리 필요)
👥 운영자 여러 명이 같이 보고 있을 때
• 라이브 모드를 여러 PC에 동시에 띄우는 건 OK. 100명 이상 큰 행사에서는 입장 게이트 2개로 나누고 QR 화면 2대 띄우는 게 정식 권장
• 마감 버튼이 없어졌으니 동시 클릭 충돌 자체가 사라짐
• 미확인 일괄 처리는 충돌 가능성 낮음 — 같은 데이터를 두 명이 동시에 바꾸려 하면 마지막 변경이 우선 (Last Write Wins) + audit log 기록
🚫 수업 자체가 못 열린 경우 — 휴강 Phase 3 이월

강사가 갑자기 못 오거나, 강의장 사고나 천재지변으로 수업 자체가 안 열린 경우. 부수 기능 묶음(보강 강제 토글 / 자동 알림 / 자동 재배정 / audit log)이 Phase 3로 이월. Phase 1·2에서는 운영자 임의 처리(미확인 일괄 → 출석 또는 결석)로 우회.

기본 처리
  • 회차 상태에 "휴강" 추가
  • 휴강된 수업은 출석부에서 그 칸을 빼버려요
  • "4번 수업 중 3번으로 점수 계산"이라고 표시
  • 관리자만 처리 가능, 사유 입력 필수
코스 설정 — "보강 강제" 토글
  • ON: 법정의무교육 — 보강 안 하면 수료 처리 불가
  • OFF: 자율 교육 — 휴강은 출석부에서 빼고 진행
  • 회차선택형은 학습자에게 다른 날짜(회차) 자동 안내
⏸ Phase 2 이월 — 자동화·시스템 잠금·알림

Phase 1은 의도적으로 단순화했습니다 — 회차 종료(시간 자동) + 운영자 수동 정리만. 아래 항목은 모두 Phase 2 이월이며, Phase 1 안정화 후 도입합니다.

⏰ 자동 처리
  • 미확인 자동 결석 처리 (+7일)
  • 시스템 잠금 — 회계 마감 (+30일)
  • 코스별 "자동 결석 처리 OFF" 토글
📧 알림
  • 운영자 이메일 4단계 (24h/72h/6일/7일)
  • 학습자 결석 확정 통보 (C3, 7일 지연)
  • HR 시스템 연동 (C4)
Phase 1 운영 가이드: 운영자가 수업 종료 후 직접 미확인 인원을 정리하세요. "미확인 일괄 처리" 도구로 빠르게 가능합니다.
용어 정리: "회차 종료"로 통일. 학습자가 "회차 차감"이라고 부르는 건 같은 의미예요 (별도 사건이 아니라 수업 종료 시각이 지나는 것). 수업 자체가 못 열린 건 "휴강"입니다 ("회차 취소"는 신청 취소와 헷갈리니 사용 금지).

4.5 수강 신청

학습자가 직접 신청하는 자율 수강과 HRD 담당자가 부서·직급 단위로 배정하는 지정 등록, 두 가지 방식을 코스별로 선택합니다.

🙋 자율 수강
학습자가 목록에서 원하는 차수를 직접 신청
• 신청 기간 내에만 가능
• 잔여석이 있어야 신청 가능
• 취소 가능 기한은 코스에서 설정
👤 지정 등록
HRD 담당자가 부서/직급 단위로 직접 배정
• 법정의무교육 등에 주로 사용
• 그룹 선택 시 정원 자동 계산
• 취소는 관리자만 처리 가능
상황별 처리 기준
정원 마감
"마감" 배지로 표시됩니다. 다른 차수 신청을 안내하며, 대기 신청은 현재 지원하지 않습니다.
재수강
코스에서 "재수강 허용" 토글을 켜야 가능합니다. 법정의무교육은 반드시 켜야 합니다.
취소 기한 초과
학습자가 직접 취소할 수 없습니다. HRD 담당자에게 요청해야 합니다.
차수 변경
현재는 기존 차수 취소 후 새 차수 신청으로 처리합니다. 직접 변경 기능은 Phase 2에서 추가 예정입니다.

15. 수강신청 3모드 화면 흐름

course-detail.html은 enrollMode에 따라 UI 분기. 바텀시트 선택 → screen-confirm → 신청 완료 모달의 공통 흐름을 공유하되 모드별 세부 UI가 다르다.

15.1 모드 정의
모드enrollMode 값설명
회원이 차수 1개 선택single등록된 차수 중 1개만 선택. 차수를 "회차 선택형"으로 두면 회차(날짜) 1개 선택
회원이 여러 차수 선택multi여러 차수를 함께 선택(최대 2개). 일정이 겹치는 차수는 함께 선택 불가 — 빌더 발행 단계에서 보장
관리자 지정 등록assigned학습자가 선택 불가, 관리자가 배정
회차 선택형sessionSelect차수 내 회차(날짜)를 학습자가 선택 — 단일 선택형의 하위 옵션(토글). 법정의무교육 등 같은 내용을 여러 날짜에 열 때: 차수 1개 + 회차 여러 개로 운영(날짜마다 차수를 따로 안 만들어도 됨). 단, 다중조합(multi)과 함께 쓸 수 없음 — 아래 상호배타 규칙 참고
회차 선택형 ↔ 다중조합 상호배타 규칙 (v1.6, 2026-06-01)
회차 선택형(sessionSelect)은 단일 선택형(single) 전용입니다. "회원이 여러 차수 선택"(multi)과는 동시에 쓸 수 없습니다.
  • 이유: 다중에서 차수별 회차 분기가 필요하면 차수를 쪼개서 표현하면 됨(기능 중복). 공존 시 학습자가 "여러 차수 + 차수마다 회차"를 2중 선택해야 해 혼란.
  • 빌더 동작: 다중으로 전환하면 회차 선택형이던 차수는 자동으로 전체일정으로 바뀌고 안내(토스트)가 뜸. 다중 상태에선 수강유형 모달의 "회차 선택형" 카드가 잠김(사용 불가).
  • 차단 위치: 학습자 화면이 아니라 관리자 빌더 입구에서 차단(잘못된 설정을 애초에 못 만들게).

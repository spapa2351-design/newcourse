💬 Phase 분류 근거
  • Phase 0 분리: 아키텍처를 Phase 1에 묶으면 일정이 무너짐. 데이터 모델·인프라는 사전 작업으로 분리.
  • Phase 1 한계: "운영팀이 손으로 하던 1사이클 처리"가 기준. 자동화·알림은 의도적으로 제외.
  • Phase 2 P1/P2 분리: 8주 한 번에 못 함. 운영팀 클레임 빈도순으로 4주씩 끊어 진행.
  • Phase 3 = 도메인 확장: 푸시 알림 인프라가 여기서 도입돼 Phase 2.2 알림 채널과 연결.
  • Phase 4 = 12개월 후: 다년간 데이터가 쌓여야 의미 있음. Phase 1 시작 후 1년 뒤 가능.
  • "마감" 개념 제거: Phase 1 spec은 시간 기반 자동 종료를 전제로 작성.
📖 이 문서 사용 가이드
  • 일정 산정: Phase별 기간 견적은 백엔드 4 + 프론트 4 = 8명 + 기획 1 + 디자인 1 기준. 팀 규모에 비례 조정 (단, 인원 늘어도 커뮤니케이션 비용으로 선형 단축은 안 됨).

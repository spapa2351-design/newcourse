# newcourse 한눈에 — 팀 공유용 쉬운 버전

> 모든 팀(기획·운영·QA·개발·CS·세일즈)이 같은 그림을 보고 시작하기 위한 요약 문서.
> 전체 기획설계서는 `newcourse-spec.md` / `newcourse-spec.html` 참조.

---

## 한 줄로

> **터치클래스에 "에디터 안 들어가도 만들어지는 새 코스 방식"이 하나 추가됩니다. 기존 코스는 그대로 두고, 새 방식으로는 영상·시험·설문·문서·출석을 블록처럼 묶어 코스를 구성합니다.**

## 왜? (5개)

1. **기존 코스는 잘 돌아가니까 안 건드립니다.** 매출 주력이라 흔들면 위험.
2. **새 방식은 에디터에 안 들어가도 코스가 만들어집니다.** 영상 올리고 끝.
3. **영상 하나만, 시험 하나만, 설문 하나만으로도 코스가 됩니다.** 지금은 못 했던 코스 유형.
4. **오프라인 출석이 같은 코스 안에 자연스럽게 들어갑니다.** 블렌디드 운영 가능.
5. **진도율이 더 이상 흔들리지 않습니다.** 콘텐츠 편집해도 수강 중인 사람 진도율 그대로.

## 그림 — 이렇게 달라집니다

### 기존 (레거시)
```
┌──────────────────────────────────────┐
│ 코스 만들기                            │
│  ├─ 저작도구 들어감                    │
│  ├─ 페이지 1 만들기                    │
│  ├─ 페이지 2 만들기                    │
│  ├─ 페이지 3 만들기                    │
│  ├─ ...                              │
│  ├─ 저장 후 코스에 연결                 │
│  └─ 수료 조건 설정 (페이지 단위)         │
└──────────────────────────────────────┘
     │
     ▼
 학습자: 페이지 묶음을 차례대로 봄
 진도율: 완료 페이지 ÷ 전체 페이지

 ⚠ 문제: 페이지 추가하면 진도율 일괄 하락
 ⚠ 문제: 영상만 올리는 단순 케이스도 저작도구 거쳐야 함
 ⚠ 문제: 오프라인 출석은 별도 시스템
```

### newcourse
```
┌──────────────────────────────────────┐
│ 코스 만들기                            │
│  ├─ 영상 / 시험 / 설문 / 자료 / 출석    │
│  │    중에서 골라서 슬롯에 넣기          │
│  ├─ 순서 정하기                         │
│  └─ 수료 조건 자동 (슬롯 단위)           │
└──────────────────────────────────────┘
     │
     ▼
 학습자: 슬롯을 차례대로 학습
 진도율: 완료 슬롯 ÷ 전체 슬롯

 ✓ 콘텐츠 편집해도 진도율 안 흔들림
 ✓ 슬롯 1개 코스도 가능 (시험만, 설문만, 영상만)
 ✓ 오프라인 출석도 슬롯 한 종류
```

## 무엇이 비로소 가능해지나 (USP)

코스 유형은 **3종**으로 통합 (2026-05-18 정론):

| 코스 유형 | 어떻게 |
|---|---|
| **콘텐츠 코스** | 영상·문서·시험·설문 슬롯 조합. **슬롯 1개짜리(영상 1개·시험 1개·설문 1개) 단독 에셋 코스도 이 유형으로 표현** |
| **오프라인 코스** | 출석 슬롯만으로 구성 |
| **블렌디드 코스** | 온라인 + 오프라인 혼합 (영상 + 출석 + 시험 등) |

> 코스 유형은 빌더 진입 시점에 확정 (`online-builder` / `offline-builder` / `blended-builder`). 빌더 안에서 다시 selector로 묻지 않습니다.

## 레거시 vs newcourse 비교

| 항목 | 기존 (레거시) | newcourse |
|---|---|---|
| 코스 만드는 방식 | 저작도구 → 페이지 묶음 | 슬롯 골라서 배치 |
| 진도율 단위 | 페이지 | 슬롯 |
| 콘텐츠 편집 영향 | 진도율 흔들림 | 진도율 안전 |
| 오프라인 출석 | 별도 시스템 | 슬롯 한 종류 |
| 영상만 코스 | 어려움 | 자연스러움 |
| 운영 부담 | 높음 | 낮음 |
| 기존 고객사 | 그대로 운영 | 신규 영업만 |
| 신규 고객사 | 선택 가능 | 선택 가능 |

> **두 시스템은 한 코스 안에서 데이터가 섞이지 않습니다.** 고객사는 둘 중 하나를 선택.

## 팀별 영향 — 이 변화가 우리 팀에 어떤 의미인가

### 기획팀
- 기획문서 단위가 페이지 → 슬롯(에셋 + 수료 조건)으로 바뀜
- 코스 유형 3종 정의 + 슬롯 6종 명세 산출물
- 수강 등록 채널 3종(A/B/C) 정의 — 현재 운영 4개 케이스 통합 처리
- 미해결 결정 안건 협의

### 운영팀
- CS 매뉴얼 두 벌 운영 (레거시 / newcourse)
- 영업 매트릭스 안내 매뉴얼 작성
- 등록 채널 3종 운영 가이드 (관리자 수동 / 자율 신청 / 가입 시 자동)
- 기존 고객 이전 요청 답변 스크립트 필요
- 차수(회차) 정식 요구 케이스 수집 (정기 갱신·동기 학습·정원 제한 등)

### QA팀
- **회귀 테스트 시나리오**가 새로 필요:
  1. 슬롯 단위 진도율 결정성 (콘텐츠 편집해도 진도율 불변)
  2. 커리큘럼 버전 동결 (v1 등록자는 v2 영향 없음)
  3. 등록 채널 3종 (각 채널로 들어온 등록자의 동작 일관성)
  4. 개인별 수강 기간 (등록일 기준 계산 정확성)
  5. (Phase 3) EnrollmentPlan 면제 케이스
- 정합성 100% 목표 (편차 0)

### 디자인팀
- **빌더는 v1 단일 화면 구조 유지 결정** — 4영역 분리는 데이터 모델 차원
- 3개 빌더 진입 (online / offline / blended) — newcourse-hub에서 선택
- "기능 적음 + 단단함" 톤의 디자인 시스템

### 개발팀
- **4객체 분리** 데이터 모델 (Course / Curriculum / Slot / EnrollmentPolicy + Enrollment)
- **차수(Cohort) 객체는 Phase 1 미포함** (정론)
- 3-layer 분리 (Presentation / Evidence / Fulfillment)
- Evidence Store 데이터 파이프라인
- 등록 채널 3종 (A: 관리자 수동 / B: 자율 신청 / C: 가입 시 자동 + 그룹 매핑)
- 개인별 수강 기간 (등록일 기준 계산)
- 멀티 디바이스 동기화 (한번 true → 그대로 true)
- 멱등성 (event_id 기반 dedup)

### CS팀
- 첫 질문: "어떤 코스 시스템인가요?" — 매뉴얼 표지에 박을 것
- 매뉴얼 두 벌 (레거시 / newcourse) — newcourse는 훨씬 얇아짐

### 세일즈팀
- 신규 영업 시 두 옵션 비교 자료
- "보수적 고객일수록 newcourse" 가설 활용
- 기존 고객 이전 요청 = 별도 컨설팅 프로젝트 (유상 검토)

## 자주 묻는 질문 (FAQ)

### Q1. 기존 고객사는 강제로 옮겨야 하나요?
A. 아닙니다. 기존 고객사는 그대로 레거시에서 운영 지속합니다. 옮기고 싶다는 요청이 오면 별도 컨설팅 프로젝트로 검토합니다.

### Q2. 한 코스 안에 두 시스템 데이터를 섞을 수 있나요?
A. 안 됩니다. 한 코스 = 한 시스템. 같은 고객사가 두 시스템을 동시에 쓰는 것도 권장 안 합니다.

### Q3. 신규 계약은 무조건 newcourse인가요?
A. 아닙니다. 두 옵션 다 가능합니다. 다만 영업 매트릭스에 따라 권장이 갈립니다.

### Q4. 레거시는 언제 없어지나요?
A. 미정입니다. 결정될 때까지 두 시스템 병행 운영합니다.

### Q5. 진도율이 안 흔들린다는 게 무슨 뜻인가요?
A. 슬롯 단위로 카운트하기 때문에, 슬롯 안의 콘텐츠(영상·문서)를 편집해도 슬롯 개수가 안 바뀌면 진도율도 안 바뀝니다. 슬롯 자체를 추가/삭제하면 새 커리큘럼 버전이 발행되어, 진행 중 수강생은 기존 버전 그대로 유지됩니다.

### Q6. EnrollmentPlan이 뭔가요?
A. 학습자별 맞춤 동결 (Phase 3 예정). 휴직 복귀자에게 일부 슬롯 면제, 특정 학습자에게 대체 슬롯 부여 같은 운영 케이스를 표현하는 장치입니다.

### Q7. Evidence Store는 뭔가요?
A. 학습 중 발생한 사실(시청 시간·페이지 열람 등)을 저장하는 곳. 나중에 진도율·수료 기준이 바뀌어도 이 데이터로 재계산할 수 있게 분리해둡니다.

### Q8. 오프라인 코스 단독으로도 가능한가요?
A. 가능합니다. 출석 슬롯만으로 구성된 코스가 가능합니다 — 별도 시스템 없이.

### Q9. 시험만 코스는 어디다 쓰나요?
A. 신입사원 입사 시 단독 평가, 법정 의무교육 후 사후 시험 등. 지금은 시험을 코스 안에 끼워 넣어야 하지만, newcourse에선 시험 1개로 한 코스(콘텐츠 코스 유형의 하위 케이스)가 됩니다.

### Q10. 진행 중 코스에 콘텐츠 추가하면 어떻게 되나요?
A. 슬롯 추가 = 새 커리큘럼 버전 발행. 기존 등록자는 등록 시점의 발행 버전 그대로 유지되고, 신규 등록자에게만 새 버전이 적용됩니다. 진도율 사고 없음.

### Q11. 차수(기수) 개념은 왜 없나요?
A. 기업 LMS는 학습자 변동·정책 변화가 잦아 대학교 LMS식 동기 학습 차수가 부적합합니다. 현재 운영 케이스 4가지 모두 차수 없이 개인별 등록 + 개인별 수강 기간으로 처리됩니다. 정기 갱신·동기 학습·정원 제한 같은 진짜 차수 케이스가 정식 요구로 올라오면 Phase 2+ 옵션으로 추가합니다.

### Q12. 수강생 등록은 어떻게 시키나요?
A. 채널 3종이 있습니다:
- **A. 관리자 수동 등록** — 명단(엑셀) 또는 그룹 선택해서 일괄 등록 (100~1000명 케이스 포함)
- **B. 학습자 자율 신청** — 코스를 신청 가능으로 열어두고, 학습자가 임의 시점에 신청
- **C. 가입 시 자동 등록** — 그룹↔코스 매핑. 신입사원 가입 시 신입 온보딩 자동 등록 같은 패턴

3개 채널을 한 코스에서 동시 활성 가능합니다.

## 일정 — 어디까지 결정됐고 무엇이 남았나

**결정된 것 (2026-05-18 정론 정렬 반영)**:
- **4객체 분리** (Course / Curriculum / Slot / EnrollmentPolicy + Enrollment)
- **차수(Cohort) Phase 1 제거** — 개인별 등록 + 개인별 수강 기간으로 정론
- **코스 유형 3종** (콘텐츠 / 오프라인 / 블렌디드) — 단독 에셋은 콘텐츠 하위
- **등록 채널 3종** (관리자 수동 / 자율 신청 / 가입 시 자동)
- 4영역 분리는 데이터 모델·API 책임 분리 차원만 적용 — UI는 빌더 v1 단일 화면 유지
- 슬롯 6종 (video / page / quiz / survey / offline_attendance / external_link)
- 커리큘럼 버전 동결 + 등록자 단위 발행 버전 스냅샷
- 운영 정책 방어선 (삭제 금지·비활성화·순서 제한)
- 레거시·newcourse 동등 옵션, 마이그레이션 미지원

**남은 결정 (다음 회의)**:
- 슬롯 완료 조건 기본값 정밀도
- 단독 에셋 코스 수료증·기록 기준
- EnrollmentPlan 권한 (운영자 자유 / 사유 / Admin 승인) — Phase 3
- 진행 중 등록자에게 슬롯 추가 요청 처리
- 익명화 기준
- 채널 C(가입 시 자동 등록) 트리거 시점 (가입 즉시 vs 그룹 매핑 완료 시점)
- 코스 만료 시 진행 중 등록자 처리 (자동 해제 vs 학습 완료까지 유지)
- 차수(회차) 정식 요구 케이스 수집 — 어떤 코스에서 동기 학습·정원 제한이 실제로 필요한지
- PDF 슬롯 → page sub-format 흡수 여부
- pageSequenceLock 의미 명확화
- 다대다 도입 시점 (Phase 3)
- Learning Graph 도입 시점 (Phase 4)

# 2. 코스 개설/편집 빌더 (admin1.html)

## 레이아웃
- 상단 헤더: ← 목록 돌아가기 + 코스명 + 상태 뱃지(초안/임시저장/퍼블리싱됨) + 임시저장 버튼 + 미리보기 버튼
- 좌측 메인(800px): 섹션 1~3
- 우측 사이드바(500px): 커리큘럼 관리 패널
- 하단 고정: 코스 퍼블리싱 버튼 (사이드바 내)

## 섹션 1: 코스 기본 정보
**상태: `courseInfo`**
- title, category, instructor(=대표 강사), instructorDesc, intro, location, address, traffic
- sessionInstructorMode (boolean, 기본 false) — "차수·회차별 강사 다름" 토글 상태
- (차수 강사·회차 강사는 각각 option.instructor / session.instructor 필드. 강사 풀 instructorPool은 폐기)
- credit (선택, number) — 학점 부여 코스 여부. C사 케이스 등 인사 연동 시 필요. Phase 1에서는 UI 노출 여부 미정 → 데이터 필드만 예약

**UI**: 코스명, 상단 표시 방식(기본표지/이미지), 카테고리, **강사 묶음 카드**(담당 강사=대표, 강사소개, "차수·회차별 강사 다름" 토글 — 입력칸만, chip 편집기 없음), 코스소개, 장소설정(장소명/주소검색/교통안내)

> **[강사 3단 상속 2026-06-08]** 강사가 차수마다(평일반/주말반), 회차마다(특강·안전보건) 다를 수 있다. 강사 레벨 = **코스=대표 강사(필수) / 차수=차수 강사(선택) / 회차=회차 강사(선택)**. 회차 비면 차수강사, 차수강사 비면 대표강사 상속. 토글 OFF면 현행(대표 1명), ON이면 세 레벨 모두 **텍스트 직접 입력**(강사 풀·콤보 폐기). 차수강사 칸은 차수 2개 이상일 때만 노출. 빈 값은 상위 상속. 출석·수료 무관(표시 메타). 상세: `docs/02-design/features/offline-session-instructor.design.md`

> **[블렌디드 검토 추가 2026-04-12]** 학점(Credit) 필드: C사처럼 수료 시 학점을 부여하는 케이스 존재. Phase 1 구현 시 UI에 표시 안 하더라도 스키마에는 `credit: number | null` 컬럼 예약 권장.

## 섹션 2: 모집 및 수강 신청 설정

### 수강 신청 기간 (항상 고정, 섹션 최상단)
- **결정**: 오프라인 코스에 무제한 개념 없음 → 자율/강제 구분 없이 항상 표시
- 신청 오픈/마감 datetime-local

### 수강 등록 방식
1. **자율 수강** (`isSelfEnroll=true`)
   - 신청 권한 그룹 지정
   - 옵션 선택 규칙: 단일 선택형(택1) / 다중 조합형(최대 N개)
   - **다중 조합형**: 최소 제거, 최대만 유지
   - **회차선택형 ↔ 다중조합 상호배타 (A안, 2026-06-01)**: 회차선택형(`sessionSelect`)은 단일 선택형 전용 — 다중 조합형과 함께 쓸 수 없음. 다중 전환 시 회차선택형 차수는 전체일정으로 자동 전환 + 토스트 안내, 다중 상태에선 수강유형 모달 "회차 선택형" 카드 잠금. 다중에서 회차 분기는 차수 분리로 표현. 빌더 입구에서 차단

2. **관리자 지정 등록** (`isSelfEnroll=false`)
   - 취소·변경 정책 UI 숨김 (관리자가 직접 처리하므로)
   - 패키지에 그룹 칩 배정 UI (정원 자동 계산)

### 취소·변경 정책 (자율 수강 전용)
**상태**: `cancelPolicy`, `changePolicy`

**수강 취소 정책** (토글 ON/OFF):
- OFF → 신청 마감일 전까지 언제든 취소 가능
- ON → 라디오 2택:
  - **기한부 취소 허용**: "교육 시작 N일/시간 전까지 가능" (기한 후 관리자 승인 필요)
  - **취소 불가**: 관리자만 취소 처리 가능

**일정 변경 제한** (토글 ON/OFF):
- OFF → 신청 마감일 전까지 옵션 변경 가능
- ON → "교육 시작 N일/시간 전까지 변경 가능" (amber 색상)

**결정**: 특정 일시 지정 방식 삭제 → 교육 시작 기준만 유지 (단순화)

## 섹션 3: 선택 옵션(패키지) 생성

### 데이터 모델: `options` 배열

> **[용어 충돌 경고 2026-04-12]** 현재 "패키지"는 코스 내 반/기수(A반, 1기)를 의미하지만, C사 케이스에서 "패키지 코스"는 여러 코스를 묶은 번들을 뜻함. C사 구조 설계 전 `패키지 → 기수` 또는 `반`으로 명칭 변경 검토 필요.

```js
{
  id: number,
  name: string,          // 옵션명 (= 기수/반명. 예: "A반", "1기")
  capacity: number,      // 정원 (그룹 있으면 자동계산)
  enrolled: number,      // 신청 인원
  startDate: string,     // 패키지 시작일
  endDate: string,       // 패키지 종료일
  place: string,         // 개별 장소 (비어있으면 기본 장소)
  groups: [{ id, name, count }],  // 지정 등록 시 배정 그룹
  sessions: [            // 세부 회차 배열 — 각 회차가 곧 하나의 오프라인 슬롯
    {
      id: number,
      slot_id: string,   // ⭐ 슬롯 고유 ID (completion_event 기록 기준, Phase 3 연결 열쇠)
      name: string,      // 회차명
      date: string,
      startTime: string,
      endTime: string,
      place: string,
      instructor: string  // 회차 강사(선택). 빈 값이면 차수강사 → 대표강사 상속. sessionInstructorMode ON일 때만 입력/표시
      // ※ 차수(option)에도 instructor: string (차수 강사, 선택). 차수 2개 이상일 때만 입력칸 노출
    }
  ]
}
```

> **[블렌디드 완료 이벤트 설계 2026-04-12]** 출석 완료를 `attendance { session_id, user_id, status }` 로 기록하면 Phase 3에서 영상/시험 완료와 합산 불가. 대신 `completion_event { slot_id, user_id, type: 'attendance', created_at }` 구조로 기록해야 블렌디드 Phase 3 확장 시 `type: 'video_end' | 'exam_submit'` 추가만으로 통합 진도 계산 가능.

### 패키지 카드 기능
- 인라인 이름 수정, 정원 입력 (그룹 있으면 자동 → 읽기전용 + "자동" 뱃지)
- 시작/종료 날짜, 개별 장소 필드
- 지정 등록 모드: 그룹 칩 UI (드롭다운으로 추가/X로 제거, 합산 표시)
- 복사, 삭제 버튼

## 퍼블리싱 플로우

### 코스 상태: `courseStatus`
- `draft` → `saved` → `published`
- 상태 뱃지 (헤더에 색상별 표시)
- 마지막 저장 시각 표시

### 임시저장
- 언제든 클릭 가능, courseStatus → 'saved' 전환

### 퍼블리싱 모달 (코스 퍼블리싱 버튼 클릭 시)
1. **등록 요건 확인** — 7개 항목 자동 검증 체크리스트:
   - 코스명 입력, 담당 강사, 교육 장소, 신청 기간, 최소 1개 패키지, 모든 패키지에 회차 존재, 모든 회차 날짜/시간 설정
2. **공개 시점 선택** (모든 요건 충족 시만 노출):
   - 즉시 공개 / 예약 공개 (datetime picker)
3. **경고 안내**: 퍼블리싱 후 신청 시작, 수정 시 알림, 삭제 시 자동 취소

### 출석 상태 5종 (2026-04-09 확정)
| 상태 | 키 | 발생 조건 | 최종 상태 여부 |
|---|---|---|---|
| 미확인 | `pending` | 회차 생성 시 전원 기본값. **임시 상태** | ✕ (반드시 다른 상태로 전환) |
| 출석 | `attended` | QR 스캔(시작 시간 내) 또는 관리자 수동 처리 | ○ |

### 출석인정(excused) 제거 이유
- 관리자가 수동으로 "출석" 처리하면 그게 곧 출석인정
- 6종→5종 단순화로 관리자 혼란 감소

### 미확인(pending) 정책
- **정의**: "아직 판단 전" 임시 상태, 최종 상태가 아님
- **라이프사이클**: 회차 생성(전원 미확인) → 교육 중(QR 스캔→출석/지각) → 수업 종료 시각 도래(회차 자동 종료) → 운영자가 미확인 인원 직접 정리
- **미확인이 남는 케이스**: ①교육 시작 전, ②QR 안 찍은 수강생, ③QR 오류/폰 없음, ④관리자 확인 중
- **회차 종료 (시간 기반 자동)**: 수업 종료 시각 = 회차 자동 종료 시각. 별도 마감 버튼 없음. 미확인 인원은 운영자가 "미확인 일괄 처리" 도구로 직접 정리
- **자동 결석 처리·운영자 단계별 알림·시스템 잠금 정책**: Phase 1엔 없음 (Phase 2 이월)
- **학습자 알림**: Phase 1은 즉시성만 (출석 처리됨, 출석률 갱신). 결석 통보·HR 연동은 Phase 2 이월

### 출석 관리 UX
- **미확인 상태**: [출석] 원클릭 버튼 + [...] 드롭다운(지각/결석) 병렬 제공
- **확정 상태**: 드롭다운으로 상태 변경 (출석/지각/조퇴/결석/미확인)
- 이전 "출석인정" 버튼 → "출석" 버튼으로 변경 (혼동 제거)

### QR 자동 갱신
- **30초 주기**
- 10초: 너무 짧아 스캔 실패 / 60초: 대리촬영 공유 위험
- QR에 타임스탬프+회차ID 포함 → 1회성 처리

### 관리자 TOP 5 액션 (우선순위)
1. QR 출석 화면 띄우기 (매 회차 시작)
2. 미출석자 확인 및 연락
3. 개별 상태 변경 (지각/조퇴/결석)
4. 회차별 출석 현황 요약 확인
5. 회차 자동 종료 (수업 종료 시각). 미확인 인원은 운영자가 직접 정리

### 패키지 전환
하루 1~2개가 일반적 → 사이드바 드롭다운이면 충분

# 용어 체계 & 아키텍처

## 용어 확정: 패키지 → 차수

**배경**: "패키지(Package)"는 개발 용어 느낌이 강하고, 현업 교육담당자에게 낯섦.
**기준**: 이러닝 현업에서 실제로 쓰는 단어, "회차"와 계층이 명확히 구분될 것.

### 후보 비교

| 후보 | 현업 사용 빈도 | 회차와 계층 명확성 | 단기·장기 모두 적합 |
|------|:---:|:---:|:---:|
| **차수** | 최상 | 코스→차수→회차 (완벽) | O |
| 기수 | 보통 | 가능 | 장기 프로그램에만 자연스러움 |
| 반 | 높음 | 가능 | 동시 운영(공간 분리)에만 어울림 |
| 회(回) | 보통 | 회차와 혼동 위험 | 보통 |

### 확정: 차수(次數)

"이 과정 3차수로 운영할 거예요?" — 현업 교육담당자가 실제로 쓰는 말
"1차수 정원 30명, 2차수 정원 20명" — 자연스러운 UI 레이블
차수 > 회차 계층이 언어적으로 완벽하게 분리됨

## 아키텍처 결정: 커리큘럼 중심 계층 구조

### 핵심 인사이트 (2026-04-12)

오프라인 출석뿐 아니라 온라인 영상·시험·설문도 결국 커리큘럼 안에서 동등하게 세팅된다.
블렌디드(Phase 3)에서 학습 흐름을 구성하는 중심축이 "커리큘럼"이기 때문.

- 커리큘럼 = 학습 흐름 구성 (무엇을 어떤 순서로) → 내용 레이어
- 코스 = 운영 관리 단위 (제목, 강사, 분류) → 메타데이터 레이어
- 차수 = 오프라인 슬롯의 실행 스케줄 → 운영 레이어

### 확정 계층 구조

코스 (무엇을 배우나 — 제목·강사·분류·신청 설정)
 └── 커리큘럼 (어떻게 배우나 — 슬롯 순서/구성)
       └── 슬롯 (개별 학습 단계)
             ├── 온라인 영상 슬롯   → 에셋 라이브러리 참조
             ├── 오프라인 출석 슬롯
             │     └── 차수 (운영 회기: 날짜·정원·그룹)
             │           └── 회차 (개별 수업일)
             ├── 온라인 시험 슬롯   → 에셋 라이브러리 참조
             └── 온라인 설문 슬롯   → 에셋 라이브러리 참조

### Phase별 커리큘럼 구성

| Phase | 커리큘럼 구성 |
|-------|-------------|
| Phase 1 (현재) | 오프라인 출석 슬롯만 → 차수 → 회차 |
| Phase 2 | 온라인 영상·시험·설문 슬롯 추가 |
| Phase 3 | 오프라인 + 온라인 혼합 슬롯 (블렌디드 완성) |

### 용어 정의 (확정)

| 용어 | 의미 | 비고 |
|------|------|------|
| 코스 | 교육 프로그램 단위 | 제목, 강사, 카테고리, 신청설정 포함 |
| 커리큘럼 | 학습 흐름 구성 (슬롯 인덱싱) | admin1 우측 사이드바 패널명으로 이미 사용 중 (올바른 이름) |
| 슬롯 | 개별 학습 단계 | 영상/출석/시험/설문 타입 |
| 차수 | 오프라인 슬롯의 운영 회기 | "몇 차수로 운영하나" (구 패키지) |
| 회차 | 차수 내 개별 수업일 | "며칠에 걸쳐 교육하나" |

주의: "커리큘럼"은 콘텐츠 구성 개념이므로 관리 단위 최상위(=코스)와 혼동하지 않는다.
admin1 우측 사이드바의 "커리큘럼 관리" 패널명은 이 정의에서 올바른 이름임이 확인됨.

# newcourse 통합 기획설계서 v0.3

> 본 문서는 터치클래스 브랜드 우산 아래의 신규 코스 시스템(newcourse)에 대한 기획설계서다.
> 레거시 시스템은 별도로 운영되며, 본 문서는 newcourse에 한정한다.
>
> **v0.3 (2026-05-18) 주요 변경**: 차수(Cohort) Phase 1 제거 · EnrollmentPolicy 신설 · 등록 채널 3종(A/B/C) · 코스 유형 6종 → 3종 · 4영역 분리는 데이터 모델만 (UI는 빌더 v1 단일 화면 유지)

---

## 한 줄 룰

> **터치클래스는 레거시와 newcourse를 동등한 옵션으로 운영한다. newcourse는 "에디터 없는 깔끔한 운영" + "새로운 코스 유형(오프라인·블렌디드·단독 에셋)이 비로소 가능"한 그린필드 시스템이며, 두 시스템 간 코스·데이터 공존은 없다.**

## 이 결정의 이유 5가지

1. **레거시는 비즈니스 주력이다 — 흔들지 않는다.**
   당장은 매출 비중이 레거시에 있다. 신규 모델을 위해 레거시를 수정·확장하면 리스크가 너무 크다. 레거시는 유지 운영(Maintenance Mode)으로 가고, 신규 기능은 newcourse에만 추가한다.

2. **newcourse의 진짜 가치 = 새 코스 유형이 비로소 가능해진다.**
   "에디터 걷어내기"는 수단이고, 본질은 오프라인 코스·블렌디드 코스·시험/설문 같은 **단독 에셋 코스**가 동일 모델 위에서 자연스럽게 표현 가능해진다는 것이다. 레거시 페이지 모델로는 못 만들던 코스 유형들이다.

3. **"기능 적음 + 단단함" — 보수적 고객 타깃 가설.**
   기업교육 운영 담당자 다수가 "영상만 올리고 끝내고 싶다"고 요청한다. 복잡한 저작도구는 학습 효과보다 운영 부담을 키운다. 보수적·단순 운영 선호 고객사에 newcourse의 단순성이 오히려 더 매력적이라는 가설.

4. **두 시스템 코스 공존 비용 > 분리 비용.**
   한 코스 안에 레거시 데이터와 newcourse 데이터를 섞으면 진도율 정합성 사고와 CS 부담이 폭증한다. 시스템 수준에서는 두 제품 동시 운영하되, 코스/데이터 수준에서는 완전 분리한다.

5. **그린필드 = 진도 정합성·운영 모델·UX 자유.**
   레거시 호환 책임이 없으니 슬롯 단위 진도율·결정적 정합성·등록 채널 3종·개인별 수강 기간·만료 정책을 처음부터 깔끔히 박을 수 있다.

## 0. 제품 포지셔닝 (전제)

### 0.1 브랜드 구조

```
터치클래스 (Brand)
├─ 레거시 (Maintenance Mode, 신규 영업 가능, 신규 기능 동결)
└─ newcourse (그린필드, 신규 영업 가능)
```

- 신규 계약은 양쪽 선택 가능. 동등 옵션이지만 당장 비즈니스 주력은 레거시.
- 한 고객사 = 둘 중 하나 (한 코스 안에 두 시스템 혼합 불가).
- 레거시 EOL 시점은 미정. 마이그레이션은 현재 미지원, 향후 별도 컨설팅 프로젝트로만 검토.

### 0.2 고객 분류 가설 (운영팀·세일즈 매트릭스)

| 고객 특성 | 권장 |
|---|---|
| 기존 페이지형 콘텐츠 자산 다수 보유 | 레거시 |
| 페이지 단위 추적이 핵심 요구사항 | 레거시 |
| 영상·블렌디드·단독 에셋 중심 운영 | newcourse |
| 에디터 진입장벽이 부담스러운 운영팀 | newcourse |
| "영상만 올려서 운영" 같은 단순 운영 선호 | newcourse |
| 오프라인 출석을 코스 안에 자연스럽게 섞고 싶음 | newcourse |
| 시험·설문 단독 운영 필요 | newcourse |

### 0.3 newcourse가 비로소 가능하게 하는 것 (USP)

1. **오프라인 코스 (단독)** — 출석만으로 구성된 코스
2. **블렌디드 코스 (혼합)** — 온라인 + 오프라인이 한 코스 안에 자연스럽게
3. **단독 에셋 코스** — 시험만 / 설문만 / 영상 1개만으로 구성된 코스
4. **콘텐츠 코스** — 영상·문서·혼합형
5. **(향후)** 라이브 세션, 과제 제출, SCORM/xAPI 등

핵심: **모두 동일한 슬롯 시스템 위에 표현 가능**. 별도 모델·별도 화면·별도 통계 없음.

## 1. 현재 솔루션 As-Is 분석 — 안티패턴 카탈로그

> 레거시 페인을 newcourse 설계의 회피 대상으로 카탈로그화한다.
> 결정의 역방향 근거이며, 신규 모델 대응을 1:1로 매핑한다.

### 1.1 데이터 모델 결함

| 결함 | 설명 | newcourse 대응 |
|---|---|---|
| 페이지 단위 진도율 | 분모가 가변이라 콘텐츠 편집이 진도율을 흔든다 | 슬롯 단위 진도율 + 슬롯 내부 콘텐츠 자유 편집 허용 |
| 코스 ⇔ 커리큘럼 1:1 임베드 | 커리큘럼이 코스의 종속 자식 = 재사용 불가 | 커리큘럼 독립 객체 + 다대다(Phase 3) |
| 회차 개념 부재 | 같은 과정을 매년 새 코스로 복제, 데이터 파편화 | 코스/회차 2계층 분리 |
| 버전 관리 없음 | 커리큘럼 수정이 진행 중 수강생에 즉시 반영 | 커리큘럼 버전 동결 + 회차 스냅샷 |
| 오프라인 별도 시스템화 | 진도율·수료 로직 두 벌 | 슬롯 타입 `offline_attendance` 하나로 흡수 |

### 1.2 운영 페인 (반복 수동 작업)

| 페인 | newcourse 대응 |
|---|---|
| 진도율 수동 재계산 | 슬롯 단위 + 버전 동결로 재계산 불필요 |
| 코스 통째 복제 + 통계 분리 | 회차 복제 + 수강생 이월 옵션 |
| 수강생 이월 엑셀 작업 | 회차 복제 시 3옵션(전체이월/초기화/일부선택) |
| 만료된 코스 식별 불가 | maxRunDays(기본 365) + 자동 아카이브 |

### 1.3 UX 페인

| 페인 | newcourse 대응 |
|---|---|
| 빌더 한 화면 과부하 | 4영역 분리 (코스 빌더 / 커리큘럼 빌더 / 콘텐츠 에디터 / 회차 관리) |
| 동일 콘텐츠 재작성 | 커리큘럼 복제·라이브러리화(Phase 3) |
| 운영자 권한 일괄 | 역할 분리 (Course/Curriculum/Cohort Manager) |

### 1.4 정합성 사고 패턴

- 페이지 추가 → 진도율 일괄 하락 (수강생 항의)
- 페이지 삭제 → 진도율 급증 (통계 신뢰도 붕괴)
- 코스 복제 시 진도 데이터 미이관
- 멀티 디바이스 동시 학습 → 진도 덮어쓰기
- 네트워크 끊김 후 재시도 → 슬롯 완료 중복 카운트 (멱등성 부재)

### 1.5 통계 결함

- 같은 과정인데 회차마다 분리된 코스 → 합산 불가
- 콘텐츠 효과성 비교 불가 (v1 vs v2 비교 데이터 없음)
- 페이지 단위 변동 때문에 시계열 신뢰성 낮음

> 운영팀·CS팀 추가 사례 수집은 별도 채널로 수렴. 본 카탈로그는 newcourse 설계 회피 대상으로만 활용.

## 2. 시스템 아키텍처 — 3-layer 분리

> 원본 2026-03 문서(방안 2)의 핵심 인사이트.
> 시스템을 명시적 3계층으로 분리해 표현·관측·판정 책임을 갈라놓는다.

```
┌─────────────────────────────────────────────────────┐
│  Presentation Layer (표현)                          │
│  "어떻게 보여주는가" — UI/UX 책임만                  │
│  비디오 플레이어 / 페이지 뷰어 / 출석 체크 UI / 퀴즈 UI │
└─────────────────────┬───────────────────────────────┘
                      │ 이벤트 발생
┌─────────────────────▼───────────────────────────────┐
│  Evidence Layer (관측)                              │
│  "무엇이 발생했는가" — 사실 기록만, 해석 없음          │
│  video_played 320s / page_3_opened / attendance_ok  │
└─────────────────────┬───────────────────────────────┘
                      │ 증거 수집
┌─────────────────────▼───────────────────────────────┐
│  Fulfillment Layer (판정)                           │
│  "이수 조건이 충족되었는가" — 결정적 판정             │
│  완료 / 진행 / 수료 / 통계 모두 여기서                 │
└─────────────────────────────────────────────────────┘
```

### 2.1 왜 이 분리가 필수인가

레거시는 세 책임이 뷰어 하나에 뭉쳐 있었다. 결과:
- 뷰어 바꾸려면 진도 로직 같이 깨진다
- 콘텐츠 타입 늘리면 통계 기준 깨진다
- 페이지 추가하면 수료 판정이 달라진다

3-layer 분리하면:
- 뷰어 교체 → Fulfillment 영향 없음
- 새 콘텐츠 타입 추가 → Evidence 어댑터만 추가
- 수료 정책 변경 → Fulfillment Layer에서 재판정 가능

### 2.2 Evidence Store 데이터 파이프라인

```
raw event  →  normalized event  →  evidence aggregation  →  fulfillment evaluation
   (로그)         (표준화)           (학습 상태 재료)            (이수 판정)
```

- 이벤트 = 로그. Evidence = 학습 상태의 재료. 둘은 다르다.
- 진도 재계산 가능, 수료 정책 변경 시 재판정 가능 — 모두 Evidence Store가 있기에 가능.

## 3. 데이터 모델 — 4객체 (정론, 2026-05-18 확정)

> **차수(Cohort) 객체는 Phase 1에 포함하지 않는다.** 기업 LMS의 변동성에 맞춰 개인별 등록(Enrollment) + 개인별 수강 기간으로 운영한다. 동기 학습·회차별 정원·강사 배정·정기 갱신이 정식 요구로 올라오면 Phase 2+ 옵션 객체로 추가.

```
Course (코스)
├─ 메타: 제목·설명·분류·강사
├─ 정책: maxRunDays(기본 365)·코스 전체 수료 기준·코스 유형(콘텐츠/오프라인/블렌디드 3종)
├─ 참조: curriculumId
└─ 상태: 초안 / 운영중 / 아카이브

Curriculum (커리큘럼) ★ 독립 객체
├─ 식별: id + version
├─ 슬롯 시퀀스: slotId × N (순서 포함)
├─ 완료 기준: 모든 필수 슬롯 완료 / N% 이상
└─ 상태: draft / published / deprecated

Slot (슬롯) ★ 진도 카운트 단위 = 에셋 + 수료 조건
├─ 타입: video / page / quiz / survey / offline_attendance / external_link
├─ Presentation 참조: viewer_type
├─ Evidence 정책: 수집 이벤트 정의
├─ Fulfillment 정책: 완료 조건
└─ 필수 여부: required / optional

EnrollmentPolicy (수강 등록 정책) ★ 신규 — 차수 대체
├─ 채널 A: 관리자 수동 등록 (명단 업로드 or 그룹 선택)
├─ 채널 B: 학습자 자율 신청 (allowedGroups[])
├─ 채널 C: 가입 시 자동 등록 (groupMapping[])
├─ 수강 기간: fixed(날짜) / days(등록일+N일) / unlimited
└─ 컴플라이언스: 시간대 제한·유휴 모니터링·서약서 (옵션)

Enrollment (등록) — 학습자 ↔ 코스 1:1
├─ userId + courseId
├─ channel (A/B/C 어느 경로로 등록됐나)
├─ enrolledAt + 개인 수강 기간 (등록일 기준 계산)
├─ 슬롯별 진도(SlotProgress)
├─ 학습 중 동결된 커리큘럼 버전 (개인 단위 스냅샷)
└─ (Phase 3) EnrollmentPlan: 학습자별 면제·대체·예외 동결
```

### 3.1 발행 버전 스냅샷 — 차수 없이 정합성 보존

회차 객체 없이도 정합성은 보존된다. 등록(Enrollment) 시점의 커리큘럼 발행 버전을 개인 단위로 스냅샷·동결한다.

```
Course Blueprint → Curriculum Published Version → Enrollment 시점 스냅샷 → (Phase 3) EnrollmentPlan
       (수정 가능)        (발행 후 불변)              (개인별 고정)            (학습자별 추가 동결)
```

운영 중인 코스에서 커리큘럼 새 버전 발행 시:
- 기존 등록자는 이전 발행 버전 유지
- 신규 등록자만 새 버전으로 시작
- 차수 객체 없이도 진도율 정합성 보존

### 3.2 EnrollmentPlan — 학습자별 동결 (Phase 3 예정)

휴직 복귀자 일부 슬롯 면제·특정 학습자 대체 노드 부여 같은 케이스에 대비.

표현 가능한 것:
- 슬롯 면제 (exempt)
- 슬롯 대체 (substitute)
- 추가 슬롯 (add — 운영자 권한)
- 완료 조건 오버라이드 (특수 사유, 운영자 권한)

## 4. 4영역 분리 (빌더 · 에디터 · 관리)

> 용어 구분
> - **빌더(Builder)** = 코스·커리큘럼을 구성·배치하는 도구 (메타·시퀀스)
> - **에디터(Editor)** = 콘텐츠 본문을 작성하는 도구 (article·exam·survey 작성)
> - **관리(Management)** = 운영·수강생을 다루는 화면

| 영역 | 다루는 객체 | 주요 책임 |
|---|---|---|
| **① 코스 빌더** | Course + EnrollmentPolicy | 메타·정책·만료·커리큘럼 연결·수강 등록 채널(A/B/C) · 코스 유형은 빌더 진입 시점에 확정(빌더 내부 selector 없음) |
| **② 커리큘럼 빌더** | Curriculum + Slot 시퀀스 | 슬롯 순서·완료 기준·버전 발행 |
| **③ 콘텐츠 에디터** | Slot 내부 | 타입·콘텐츠 본문·Evidence·Fulfillment 정책 |
| **④ 수강생 관리** | Enrollment | 등록자 명단·진도 모니터링·수동 등록/해제·(Phase 3) 학습자별 예외 처리 |

UX 숏컷: "이 코스에서 새 커리큘럼" / "이 코스에 수강생 등록" 컨텍스트 점프 제공. 데이터 모델은 분리, UX는 연결.

**결정(2026-05-18)**: 4영역 분리는 데이터 모델·API 책임 분리 차원에서 적용. UI는 빌더 v1의 단일 화면 구조 유지 — 운영자가 한 화면에서 메타·커리큘럼·운영 옵션을 모두 다루는 것이 단순 운영 가설에 부합.

## 5. 슬롯 타입 시스템

### 5.1 Phase 1 (6종)

| 타입 | 완료 조건 (기본값) | Evidence 출처 |
|---|---|---|
| `video` | 시청률 ≥ 80% (구간 누적) | 플레이어 heartbeat + watched_segments |
| `page` | 열람 + 30초 체류 | 진입/이탈 이벤트 |
| `quiz` | 정답률 ≥ 60% | 채점 결과 |
| `survey` | 제출 완료 | 설문 응답 저장 |
| `offline_attendance` | 회차 출석 상태 ∈ {출석·지각·조퇴} | 회차 출석 기록 |
| `external_link` | 클릭 + 외부 콜백 | 클릭 이벤트 + 콜백 API |

### 5.2 Phase 2 확장 후보

`assignment` / `live_session` / `discussion` / `scorm`

### 5.3 핵심 원칙

1. 완료 조건은 서버(Fulfillment Layer)에서 판정. 클라이언트 신호 재검증.
2. 완료 = 일회성 상태 전이. 한번 완료된 슬롯은 콘텐츠 변경으로 미완료 회귀 없음.
3. 슬롯 내부 콘텐츠 편집 자유. 슬롯 추가/삭제/순서 변경 = 새 커리큘럼 버전 강제.
4. 필수(required)만 진도율 카운트.
5. 타입은 plugin-style(`type` + `config` JSON) — 새 타입 추가 시 DB 스키마 변경 불필요.

## 6. 커리큘럼 버전 관리

### 6.1 상태 전이

```
draft ──[발행]──> published ──[새 버전 발행]──> deprecated
                      │
                      └──[Enrollment 생성]──> frozen-per-enrollment
```

### 6.2 등록자 단위 스냅샷 (차수 없이)

- Enrollment 생성 시 curriculum.publishedVersion을 개인 단위로 스냅샷
- 진행 중 등록자에게 새 버전 강제 적용 = 금지 (정합성 붕괴)
- (Phase 3) EnrollmentPlan으로 학습자별 추가 동결 가능

### 6.3 엣지 케이스

| 상황 | 처리 |
|---|---|
| v1 학습 중 v2 발행 | 기존 등록자는 v1 유지, 신규 등록자만 v2로 시작 |
| v2에서 slot 제거 | v1 등록자 데이터 보존. v2 신규 등록자에만 적용 |
| 슬롯 내부 콘텐츠만 수정 | 버전 발행 불필요. 즉시 반영. 진도율 무영향 |
| slot 타입 변경 | 새 버전 발행 강제 |
| 진행 중 등록자에게 새 버전 적용 시도 | UI 차단 + 명시적 경고 |

## 7. 진도율 정합성

### 7.1 공식 (서버 단일 계산, Fulfillment Layer)

```
progress(enrollment) =
  COUNT(completedSlots WHERE slot.required = true AND NOT exempt)
  / COUNT(allSlots WHERE slot.required = true AND NOT exempt)

기준 = enrollment.snapshotCurriculumVersion + enrollment.plan(Phase 3)
```

### 7.2 안티치트

| 리스크 | 대응 |
|---|---|
| 영상 스킵 후 완료 위장 | watched_segments 구간 누적 |
| 백그라운드 탭 시간 흐름 | heartbeat 5초 간격 |
| 페이지 즉시 이탈 | 최소 체류시간 30초 |
| 네트워크 재시도 중복 카운트 | event_id 기반 멱등성 |
| 퀴즈 무한 재시도 | max_attempts 정책 |

### 7.3 멀티 디바이스 동기화

- 슬롯 완료 상태: 한번 true → 그대로 true (단방향)
- 슬롯 내부 진행 상태(영상 watched_segments): union merge
- 오프라인 학습: 로컬 큐 → 온라인 복귀 시 dedup 후 반영

### 7.4 결정적 정합성 목표

동일 슬롯 구성 + 동일 학습 입력 → 진도율 결과 항상 동일. 콘텐츠 편집·페이지 변경이 진행 중 수강생 진도율에 영향 없음.

## 8. 코스 유형 — 3종 (2026-05-18 확정)

> newcourse가 비로소 가능하게 하는 코스 유형들. 별도 모델 아님. 슬롯 조합으로 표현.

| 코스 유형 | 슬롯 구성 예시 | 비고 |
|---|---|---|
| 콘텐츠 코스 (`content`) | video / page / quiz / survey 조합 | 가장 기본. 슬롯 1개짜리 단독 에셋(영상 1개·시험 1개·설문 1개)도 이 유형 |
| 오프라인 코스 (`offline`) | offline_attendance 슬롯 N개 | 출석만으로 구성 |
| 블렌디드 코스 (`blended`) | video + offline_attendance + quiz 혼합 | 자연스러운 혼합 |

핵심:
- **진도율·수료·통계 로직 단 1벌**. 코스 유형별 코드 분기 없음.
- **코스 유형은 빌더 진입 시점에 확정**. `online-builder` / `offline-builder` / `blended-builder` 3개 빌더가 각각 담당. 빌더 내부엔 유형 selector 노출 금지.
- 단독 에셋(시험 1개·설문 1개·영상 1개) 코스는 콘텐츠 코스 하위 케이스. 별도 유형 라벨 만들지 말 것.

## 9. 수강 등록 정책 (정론, 차수 대체)

> 차수(회차) 객체는 Phase 1엔 없다. 현재 운영 4개 케이스(관리자 일괄 강제·관리자 그룹별 강제·자율 신청·가입 시 자동)를 등록 채널 3종으로 통합.

### 9.1 등록 채널 3종

| 채널 | 트리거 | 대상 지정 |
|---|---|---|
| **A — 관리자 수동 등록** | 관리자 등록 버튼 | 명단(엑셀 업로드) 또는 그룹 선택. 100~1000명 단위 일괄 강제 등록 포함 |
| **B — 학습자 자율 신청** | 학습자 신청 버튼 | 신청 가능 그룹으로 제한 가능 |
| **C — 가입 시 자동 등록** | 회원가입 이벤트 | 가입자 그룹 ↔ 코스 자동 매핑 (예: "신입사원" 그룹 → "신입 온보딩" 코스) |

채널 A·B·C는 한 코스에서 동시 활성 가능.

### 9.2 수강 기간 (개인 단위)

| 모드 | 의미 |
|---|---|
| `fixed` | 코스에 지정된 시작·종료 일자. 모든 등록자 공통 윈도우 |
| `days` | **각 학습자 등록일 기준 N일** (개인별 만료 시점) |
| `unlimited` | 코스 자체 만료(maxRunDays) 전까지 무제한 |

### 9.3 코스 만료

- maxRunDays 기본 365일. 만료 시 코스 자동 아카이브, 신규 등록 중단.
- 진행 중 등록자 처리는 운영팀 협의 (자동 해제 vs 학습 완료까지 유지) — 미해결 논의 §16 참조.

### 9.4 차수가 필요한 케이스 (Phase 2+ 옵션)

다음이 정식 요구로 올라오면 차수 옵션 객체 신설 검토:
- 동기 학습 (함께 시작·함께 끝나는 진행)
- 회차별 정원 제한
- 회차별 강사 배정
- 정기 갱신 추적 (의무교육 만료·이월)

## 10. 운영 정책 방어선 — 정합성 2층 방어

> 기술 versioning만으로 안 되는 부분을 운영 정책으로 1차 방어.
> 원본 2026-03 방안 4(§8)의 핵심 인사이트.

### 10.1 운영 시작 후 제약

- **항목 삭제 금지** — soft delete만 (`active = false`)
- **순서 변경 제한** — 신규 슬롯은 시퀀스 뒤에만 추가
- **비활성화만 허용** — 진행 중 회차의 슬롯 비활성화는 신규 학습자에게만 반영, 기존 학습자는 EnrollmentPlan 보호

### 10.2 효과

기술 versioning 없이도 정합성 사고의 80%를 막는다. 비용 대비 효과 1등 정책.

## 11. API 책임 분리 (개념 수준)

```
/courses                          # 코스 CRUD
/courses/{id}/enrollment-policy   # 수강 등록 정책 (채널 A/B/C + 수강 기간)
/courses/{id}/enrollments         # 코스 등록자 목록
/curricula                        # 커리큘럼 CRUD
/curricula/{id}/versions          # 버전 목록 / 발행
/slots                            # 슬롯 CRUD
/groups                           # 그룹 CRUD (조직 단위)
/groups/{id}/auto-enroll-courses  # 채널 C 매핑 (그룹 ↔ 자동 등록 코스)
/enrollments                      # 등록 CRUD (채널 A/B/C 모두 진입)
/enrollments/{id}/plan            # EnrollmentPlan (면제·대체) — Phase 3
/enrollments/{id}/progress        # 진도율(서버 계산, Fulfillment Layer)
/enrollments/{id}/slots/{sid}/events  # 슬롯 이벤트(멱등성)
/evidence/{enrollmentId}/{slotId} # Evidence Store 조회
```

> 차수(Cohort) 관련 API는 Phase 1엔 없음. Phase 2+ 옵션 객체로 추가 시 별도 라인 신설.

## 17. 용어 사전

| 용어 | 정의 |
|---|---|
| Course (코스) | 콘텐츠 컨테이너. 메타 + 정책 + Curriculum 1개 + EnrollmentPolicy |
| Curriculum (커리큘럼) | 슬롯 시퀀스. 버전 단위 동결. 운영팀 언어 = "커리큘럼" (학습 순서·구성) |
| Slot (슬롯) | 진도 카운트 단위 = **에셋 + 수료 조건**. Presentation + Evidence + Fulfillment 정책 묶음 |
| Asset (에셋) | 운영팀 언어. 영상·시험·설문·문서·출석·외부 링크 등 학습 콘텐츠 자체. 개발 모델에서는 `Slot.contentRef`로 참조됨 |
| EnrollmentPolicy (수강 등록 정책) | 학습자가 코스에 어떻게 들어오는지 정의. 채널 A/B/C + 수강 기간 + 컴플라이언스 |

# 온라인 빌더 운영기간 정책 — 개발팀 1년 강제 제안 반박

> 2026-05-18 · 윗선 ↔ 기획·운영·QA·개발 Q&A

---

## 개발팀 제안 (재인용)

신규 코스에 최대 운영 가능 기간 1년 강제 지정. 데이터 누적·이관 정책 부재 해소 목적. 갱신형 운영은 별도 코스 복사 UX 제공. 만료 데이터는 별도 저장소로 archive.

---

## 한 줄 정책

> **"운영기간은 시스템 필수가 아니라 개발 구현 단순화 목적. 사용자 정책을 강제할 만큼의 필수성은 없고, 백엔드 조건 조합 로직으로 동일 효과 가능."**

---

## 검토 결론

운영기간 1년 강제 도입 보류. 현재 UX(수강기간 3종 옵션 + 종료하기 액션)으로 운영자 통제권을 유지하고, 데이터 라이프사이클은 백엔드 자동 archive 로직으로 처리.

---

## 우려 케이스 5종

1. **"신청 후 90일 수강 가능" 코스에서 운영 기간 종료 30일 전 신청** — 약속한 90일 못 채움. 신청 차단·다른 코스 안내 기능 추가 개발 필요

2. **1년 넘게 운영해야 하는 코스 생성 불가** — 예: 안전관리자 갱신교육 13개월

3. **매년 코스 복사 시 코스 ID가 매년 분리** — 작년·올해 통계 합치려면 2번 따로 뽑아야 함

4. **운영 기간 종료 후 학습자 1명만 연장 요청** — 그 사람만 따로 수강기간 연장 불가

5. **갱신형 교육 코스 복사 시 수강생 처리 딜레마**
   - 수강생 미복사 → 매년 100~1000명 재등록 작업 발생
   - 수강생 복사 가능 → 데이터 정리·archive 효과 자체가 사라짐

## 비개발자용 비유 — 아파트 모델

| 시스템 | 비유 |
|---|---|
| 코스 | 아파트 건물 |
| 학습자 | 입주자 |
| 수강기간 | 입주자 임대 계약 기간 |
| 운영기간 | 건물 운영 종료일 |

수강기간만 있으면 입주자별 만료는 명확하나, **건물 자체 archive 시점 판단**이 까다로워짐. 수강기간 옵션 중 "제한 없음" + "신청 후 N일"이면 새 입주자가 계속 들어와서 건물 끝 신호 부재.

운영기간 한 줄이 있으면 archive 판단이 단순해지나, 없어도 조건 조합으로 처리 가능: **마지막 학습자 수강기간 종료 + 신규 신청 0 + grace 90일**.

→ 운영기간 도입의 진짜 동기는 **개발 구현 단순화**. 사용자 정책 강제 명분으로는 부족.

---

## 대안 — 학습자 단위 자동 archive + 코스 자동 휴면

| 트리거 | 액션 |
|---|---|
| 학습자 1인 수강기간 종료 + grace 90일 | 해당 학습자 활동 데이터 cold storage 이관 |
| 신청기간 종료 + 모든 학습자 수강기간 종료 + 6개월 무활동 | 코스 자동 휴면 → cold storage |
| 운영자 "종료하기" 액션 (수동) | grace 90일 후 자동 archive |

---

## 신규 코스 라이프사이클 액션 — "종료하기" 4단계 분리

현재 단일 "종료하기" 버튼을 신규 코스에선 4종으로 분리.

| 액션 | 학습자 영향 | archive 트리거 | 복구 |
|---|---|---|---|
| 휴면 | 신규 신청 차단, 기존 학습자 유지 | X | 즉시 가능 |
| 정상 종료 | 신규 신청 차단, 기존은 본인 수강기간까지 | grace 90일 후 자동 | grace 내 복구 |
| 긴급 차단 | 즉시 전원 접근 차단 | X | 운영자 수동 해제 |
| archive (자동) | cold storage 이관 | 정상 종료 + grace 만료 | 별도 복원 요청 |

UX: 코스 상단 "운영 관리" 드롭다운에 4종 액션 묶기. 각 액션 클릭 시 영향받는 학습자 수·archive 시점·복구 가능 여부 표시 모달.

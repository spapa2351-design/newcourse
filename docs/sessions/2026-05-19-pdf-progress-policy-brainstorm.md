# 2026-05-19 PDF 진도율 처리 브레인스토밍 (미결정)

배경: course-viewer 작업 중 PDF 진도율 환원이 어렵다는 UX 문제 발견. 두 갈래 옵션 중 결정 필요.

## 옵션 비교

### 옵션 A — PDF를 진도율에 포함 (페이지 열람·스크롤 추적)
- 장점: 영상과 일관, 컴플라이언스 코스 PDF 의무화 가능, 학습 데이터 확보
- 단점:
  - 페이지 빠르게 넘김 회피 쉬움 → 이수 신뢰성 낮음
  - PDF.js + 스크롤 이벤트 등 구현 복잡
  - 학습자 인지 부담(정독·메모·검색 행위가 진도율로 환원 안 됨)

### 옵션 B — 자료/첨부 스타일 (진도율 제외) ★ 추천
- 장점:
  - 표준 LMS 패턴
  - 운영팀 실무 패턴(PDF 본 후 시험으로 검증)과 일치
  - 학습자 UX 자연스러움
  - 구현 단순, Phase 1 적합
  - Phase 2 옵션 C 확장 여지
- 단점:
  - 컴플라이언스 PDF 의무화 직접 지원 못함 (시험으로 우회)

### 옵션 C — 하이브리드 (운영자 선택 토글)
- 장점: 컴플라이언스/일반 모두 커버
- 단점: 빌더 옵션 증가, [[project_newcourse_single_course_scope]] Phase 1 단순성과 충돌
- 결정: Phase 1 제외, Phase 2 안건

## 추천: 옵션 B

### 적용 시 변경 사항

#### 데이터 모델
- 항목 유형: `pdf` → 표시명 "자료" (코드명은 유지 검토 또는 `material`로 리네이밍)
- `isRequired`, `progress` 필드 자료 항목에서 비활성 (UI 비노출)

#### 빌더 (`admin/online-builder-v1.html`)
- 항목 추가 모달: "자료" 라벨로 변경
- `SlotSettingModal`: 자료 항목 진입 시 이수 조건 영역 "자료는 이수 설정이 없습니다" 안내, 필수/선택 토글 숨김
- 항목 카드: 자료 항목의 필수 칩 비노출

#### 뷰어 (`user/course-viewer.html`)
- 자료 항목 카드: 진도율 칩·필수 칩·✓ 원 비노출, 📎 아이콘 + 다운로드 액션
- 사이드바: 그룹 내 학습 항목 / 자료 별도 섹션 분리 검토 (조승우 차장 안)
- `isCompleted` 함수: 자료 항목은 `true` 또는 진도율 계산에서 제외

#### `canEnterItem` lock 로직
- 자료 항목은 lock 체크 대상 제외 (필수 학습 아님)

## 기존 운영 케이스 (사용자 보완 컨텍스트)

- 에디터에 PDF 버튼 → 커리큘럼·페이지에 PDF 버튼 N개 노출
- 학습자 클릭 → PDF 뷰어 노출
- 한 커리큘럼에 여러 PDF 운영 케이스 다수
- 기존 정책: **페이지 오픈 = 진도율 100%**, 수료 영향 0

## 운영 케이스 반영 확정안 (옵션 B + 기존 정책)

1. **PDF 슬롯 1:1** — 슬롯 1개 = PDF 1개 (데이터 단순, UI에서만 묶음)
2. **클릭/오픈 시 `progress = 100`** — 기존 정책 유지
3. **수료 영향 0** — `isRequired` 강제 false, lock 체크 제외, 수료 카운트 제외
4. **자료 카드 slim 스타일** — 영상 카드보다 작게, 별도 섹션 분리 안 함
5. **항목 유형 표시명 "자료"** — 코드명 `pdf` 유지 (호환성)

## 미결정 사항 (사용자 확정 대기)

- 위 확정안 적용 OK 여부

## 적용 시 변경 영역

| 영역 | 변경 내용 |
|---|---|
| 빌더 `SlotSettingModal` | `pdf` 타입 = 커뮤니티 기능만 노출, 이수 그룹·필수 토글 숨김 |
| 빌더 항목 추가 모달 | "PDF" 라벨 → "자료" |
| 빌더 항목 카드 | 자료 슬롯의 필수 칩 비노출 |
| 뷰어 사이드바 | 자료 카드 slim 스타일, 필수 칩·진도율 칩 비노출 |
| 뷰어 PDF 뷰 | 진입 시 `progress = 100` 자동 |
| `canEnterItem` | 자료 슬롯은 차단 항목에서 제외 |
| `isCompleted`·`overallProgress` | 자료 슬롯은 완료 카운트에서 제외(또는 항상 완료로 처리) |

# 코스 라이프사이클 정책 Q&A

> 2026-05-15 · 개발자 ↔ 기획·운영·디자인·QA 문답 기록
> 대상: 신규 온라인 코스 빌더 (online-builder)

---

## Q1. 신규 코스에 "최대 운영 가능 기간"을 기본 1년으로 강제할까?

> **개발자**: 운영기간 제한이 없어서 1개 코스가 수년간 운영되며 데이터가 쌓여 시스템 부하·이관 정책 수립 불가. 1년 기본 + 매년 갱신 코스는 복사 UX로 해결하자는 제안. 터치클래스 내내 논의됐지만 고객사 관성으로 미반영됐던 사안. 보존기간이 만료된다고 데이터가 사라지는 것은 아니고, 별도 저장소로 아카이브될 예정이라 과거 이력 트래킹엔 문제 없음.

**1차 검토 결과**: 강제 1년 도입은 보류. 다음 사유로 **선택적·세분화된 정책**으로 전환.

---

## 정책 한 줄

> **"운영기간은 차수의 속성이지 코스의 속성이 아니다. 차수별 코스에만 운영기간을 강제하고, 단발성 코스는 자동 라이프사이클로 관리한다."**

---

## 왜 강제 1년이 아닌가 (거부 명분 5)

1. **시스템 부담을 고객 부담으로 전가하는 방향 오류**. 데이터 양·이관 정책은 시스템 측 문제인데, 정책 강제로 운영자에게 학습·결정 부담을 떠넘기는 구조가 됨.
2. **B2B 신뢰·영업 영향**. "이 솔루션은 1년 지나면 코스가 만료된다"를 영업이 설명해야 하는 순간, 무제한 운영을 원하는 대형 고객사가 이탈 가능.
3. **컴플라이언스 케이스 충돌**. 안전교육·법정의무는 "이수 이력 영구 보존"이 의무. 운영기간으로 코스가 끝나면 재이수·이수증 발급 체계가 오히려 복잡해짐.
4. **차수별 운영 도입으로 80% 해결됨**. 좀비 코스는 단발성 코스에서 생기는 문제. 차수별 코스는 차수마다 자연스럽게 데이터 분리 → 운영기간을 코스 전체에 강제할 필요 없음.
5. **시스템 부하의 기술적 대안 존재**. 비활성 코스 자동 인덱싱 분리, 콜드 스토리지 자동 이관, 읽기 분산 — 운영자에게 정책 부담 없이 처리 가능.

## Q2. 그럼 좀비 코스 문제는 어떻게 풀까?

**최종안: C + 단발성 자동 라이프사이클**

| 구분 | 차수별 코스 | 단발성 코스 |
|---|---|---|
| 운영기간 | **강제 (기본 1년, 연장 가능)** | 강제 X (운영자 자유) |
| 만료 시 동작 | 차수 종료 → 보존기간 → 아카이브 | 자동 라이프사이클 적용 |
| 신규 운영 단위 | "신규 차수 추가" 액션 | "코스 복사"로 별 코스 생성 |
| 누적 통계 | 차수 간 비교·합산 가능 | 코스 단위로 독립 |

### 단발성 코스 자동 라이프사이클

```
[활성] ─ 마지막 학습 활동 3개월 무음 ─→ [운영자 알림]
   ↑                                          │
   └ "계속 운영" 클릭                          │
                                              ↓
                          [추가 1개월 무대응] → [자동 비활성: 읽기 전용]
                                                       │
                                                       ↓
                                          [비활성 6개월 후] → [콜드 스토리지 이관]
```

**활동 임계값**: 학습자 1명이라도 최근 활동 있으면 활성 유지. → 운영 중 코스가 갑자기 비활성되지 않음.

## Q3. "차수별 운영"을 코스 상위 옵션으로 두는 게 맞나?

> **윗선 지적**: 운영기간 + 코스 복사 조합이 사실상 차수별 운영을 흉내내는 건데, 왜 "차수"를 1급 옵션으로 안 두냐?

**확정**: 1급 옵션으로 도입.

### 데이터 모델 (MVP 필수)

```
Course (메타)
  ├─ id, title, description, instructor, eligibility …
  └─ runMode: "single" | "cohort"

Cohort (실 운영 단위) — Course 1 : N Cohort
  ├─ id, courseId, sequence (1차, 2차…)
  ├─ opStart, opEnd, retentionDays
  ├─ studyPeriodType, studyDays, studyFixedStart, studyFixedEnd
  └─ status: "active" | "expired" | "archived"
```

**단발성 코스도 내부적으로 Cohort 1개짜리**로 처리 → 나중에 차수 추가가 단순 INSERT로 끝남. v2에서 데이터 마이그레이션 부담 0.

### UX

- 코스 등록 첫 화면 상단에 **"운영 방식" 토글** — "단발성 코스" / "차수별 코스" 2지선다 (디폴트: 단발성).
- 차수별 선택 시 운영기간 입력이 "1차의 기간"으로 컨텍스트 전환.
- 차수별 코스 메인 화면에 "신규 차수 추가" 버튼.

## Q4. 수강기간과 운영기간이 충돌하는 케이스는?

### 케이스 짚기

| 케이스 | 문제 |
|---|---|
| 대량등록 후반 추가자 | "신청 후 N일" 수강기간이 운영기간 종료를 넘김 → 자동 캡 시 광고 약속 위반 |
| 오픈코스 마지막 5일 자기수강신청 | 5일만 듣게 됨 → 컴플레인 직행 |
| 운영자가 운영기간 사후 단축 | 진행 중 수강자 학습기간 통째로 끊김 → 책임 분쟁 |
| 절대 기간(시작~종료) 옵션 | 수강기간 종료일이 운영기간 종료일 초과 시 처리 미정 |
| "제한 없음" 라벨 → 1년 강제 | 기존 고객사엔 라벨 트릭으로 비치며 신뢰 손상 |

### 정책 (6개)

1. **3단 라이프사이클**: 운영기간 → 보존기간 → 아카이브.
2. **수강기간 자동 캡 금지**: 운영기간 잔여 < 최소 수강일수면 신규 신청 차단(자동 캡 대신).
3. **절대 기간 옵션은 운영기간 범위 내로만 입력 가능** — 초과 시 인풋 자체 차단.
4. **운영자 운영기간 단축 시**: 기등록자 grace 자동 보장(원래 수강기간까지), 신규 신청만 차단, UI 영향 미리보기 필수.
5. **운영기간 종료 후 보존기간**: 읽기 전용 수강 모드(콘텐츠·이수증 열람만, 시험·과제·진도 갱신 차단).
6. **"제한 없음" 옵션 라벨 변경**: "운영기간 종료일까지(YYYY-MM-DD)"로 실효 종료일 직접 노출.

## Q5. UI가 너무 복잡해 보이는데?

**단순화 5원칙** (운영자 부담 완화):

1. **운영기간 1개만 필수 입력**. 보존기간·최소 수강일수는 시스템 기본값, "고급 설정"에서만 노출.
2. **수강기간 옵션 3→2개** ("고정 기간" / "신청 후 N일"). "운영기간 종료일까지"는 자동 계산 결과로 흡수.
3. **만료 후 정책 4개 토글 → 안내 텍스트로 대체** (못 끄는 정책이라 토글이 혼란 유발).
4. **등록 영향 시뮬레이션 → 메인에서 제거**. "영향 확인" 버튼 모달로.
5. **운영기간 단축 미리보기 → 단축 액션 시점에만 모달**. 등록 화면엔 노출 없음.

### 용어 정리

| 기존 | 변경 |
|---|---|
| 보존기간 | 종료 후 열람 가능 기간 |
| 최소 수강일수 | (운영자 화면에서 숨김, 고급 설정에만) |
| 제한 없음 | 운영기간 종료일까지 (YYYY-MM-DD) |

## Q6. 글로벌 LMS는 어떻게 처리하나? (벤치마크 5건)

| LMS | 종료 정책 | 종료 후 처리 | 운영 패턴 |
|---|---|---|---|
| **Canvas** (Instructure) | Course End Date + Term 단위 자동 종료 | "Restrict viewing after end date" 토글 — read-only/차단 선택 | 대학 표준. 학기 자동 종료 + course copy로 신규 학기 시작 |
| **Moodle** | Course End Date 필드, 강제 만료는 cron/plugin | 종료 후엔 "ended" 표시만, 접근 기본 유지 | 다수 운영자가 미설정 → 좀비 코스 문제 동일 발생 |
| **Docebo** | Validity Period (Soft/Hard Deadline) | Hard 종료 시 expired, 재이수 정책 연동 | 기업 컴플라이언스. Recurring(자동 갱신) 옵션 별도 |
| **Cornerstone OnDemand** | Training Expiration ("이수 유효기간" 중심) | 만료 N일 전 자동 알림, 재이수 강제 | 안전·법정의무 교육. 1년 단위 재이수 표준 |
| **Blackboard / Anthology** | Course Availability + End Date | 종료 시 "Course Unavailable" 자동 전환 → 30~90일 후 archive | **3단 라이프사이클이 가장 유사** (active → unavailable → archived) |

### 인사이트 4개

1. **3단 라이프사이클은 글로벌 표준** — Canvas·Blackboard 모두 active → read-only → archive 흐름. 우리 정책이 표준에 부합.
2. **"미설정 좀비 코스"는 Moodle도 동일** — 기본값 1년이 정답.
3. **컴플라이언스는 별도 모델** — Docebo·Cornerstone의 "이수 유효기간 + 재이수 강제"는 v2 백로그로.
4. **"학기/Term" 상위 묶음 개념 유용** — 우리의 "차수별 코스"가 같은 방향.

## 개발팀 답신 (확정)

> 운영기간 전면 강제 도입은 보류. **차수별 코스에서만 운영기간 강제 적용**(개발 제안 그대로 수용). 단발성 코스의 좀비 문제는 **자동 비활성 + 콜드 스토리지 이관 메커니즘**으로 대응 — 야간 배치 기반, 활동 임계값으로 운영 중 코스 보호. 시스템 부하 측면 효과는 운영기간 강제와 동등 또는 그 이상 예상. 자동 정리 로직 명세는 별도 협의 요청.

## plan 반영 항목

1. **데이터 모델**: Course(메타) + Cohort(차수) 1:N — MVP 필수
2. **"운영 방식" 토글**: 코스 등록 첫 화면 상단, 단발성/차수별 2지선다
3. **차수별 코스**: 차수마다 운영기간·수강기간·보존기간 독립 보유
4. **단발성 코스**: 운영기간 미설정 허용 + 자동 라이프사이클 (3개월 무음 → 알림 → 1개월 무대응 → 비활성 → 6개월 후 콜드 스토리지)
5. **단순화 5원칙**: 운영기간 1개 필수 + 고급 설정 접힘 + 안내 텍스트 + 영향 미리보기 모달
6. **글로벌 LMS 벤치마크**: plan 배경 섹션에 인용
7. **v2 백로그**: Recurring 코스(자동 갱신), 코스 시리즈/Term, 차수 간 수강생 이관, 시험 응시 횟수 누적, 이수증 갱신

## 운영 흐름

### 코스 등록
1. 운영 방식 선택 (단발성 / 차수별)
2. 차수별이면 → 1차 운영기간·수강기간·보존기간 입력
3. 단발성이면 → 운영기간 입력은 선택 (디폴트 1년 또는 무제한)
4. 고급 설정에서 보존기간·최소 수강일수 조정 (기본값 사용 권장)

### 코스 운영 중
- 차수별 코스: 종료 임박 시 운영자 알림 → "신규 차수 추가" 액션
- 단발성 코스: 활동 무음 3개월 시 알림 → "계속 운영" 또는 무대응 시 자동 비활성

### 코스 만료 후
- 보존기간 동안 읽기 전용 수강 모드
- 보존기간 만료 시 자동 아카이브(콜드 스토리지)
- 통계·이력은 아카이브에서도 조회 가능

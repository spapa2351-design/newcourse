오프라인 코스 아키텍처 Q&A

개발자 ↔ 기획자 문답 기록 — 대상자 구조 · 강제할당 · 차수 권한 · 출석률 정책 · 변경 정책

👨‍💻 개발자
"2026년 AI 활용 비법" 코스 대상이 전직원인데, 커리큘럼 C(오프라인)는 개발팀 한정 조건으로 생성할 수 있지 않나요? 처음에 이렇게 생각했습니다.
🙋 기획자

좋은 포인트예요. 처음에 그렇게 생각하는 게 자연스러운 방향인데, 현재 Phase 1 구조는 달라요.

대상자(수강 권한) 설정은 코스 레벨에만 있어요. 커리큘럼·슬롯 단위로 "이건 개발팀만" 같은 분기는 없습니다.

그룹 분리는 차수 레벨에서 처리해요.

오프라인 슬롯
  ├── 1차수 → 개발팀 배정
  └── 2차수 → 영업팀 배정
커리큘럼별 대상 분기는 온라인+오프라인이 혼합되는 블렌디드 코스(Phase 3) 시점에 설계할 내용이에요. 데이터 모델 설계할 때 그 확장 가능성을 막는 구조는 피해주면 좋겠어요.
👨‍💻 개발자
새 온라인 코스에서는 가능할 수도 있지 않나요? 그런데 커리큘럼별로 그룹을 강제할당하는 게 온라인 코스에서 맞는 건가요?
🙋 기획자

온라인 빌더도 수강 대상 설정은 코스 레벨에만 있어요. 커리큘럼 단위 분기는 없습니다.

그리고 "강제할당" 개념 자체가 온라인에서는 맞지 않아요.

오프라인 강제할당온라인 커리큘럼 분기
의미"이 사람들이 이 날 이 장소에 와야 해""이 사람들은 이 챕터를 봐야 해"
성격물리적 출석 배정역할 기반 필수 콘텐츠
맞는 용어강제할당Role-based mandatory content

온라인에서 같은 니즈는 학습 경로(Learning Path) 또는 그룹별 이수 조건 개념으로 풀어야 하며, 현재 미설계 영역입니다.

👨‍💻 개발자
커리큘럼별 대상 분기가 없으면 기획이 잘못된 건가요?
🙋 기획자

잘못된 게 아니에요. Phase 1 범위 내에서 의도적으로 스코핑한 거예요.

판단
Phase 1 오프라인 설계 자체✅ 맞음
차수 레벨 강제할당 개념✅ 맞음
블렌디드 대비 확장성⚠️ Phase 3 전에 슬롯 레벨 대상 설계 필요

블렌디드 코스가 되면 슬롯/커리큘럼 레벨로 대상 조건을 내려야 하는 설계 부채가 있어요. 지금은 Phase 1을 깔끔하게 끝내고, 블렌디드 기획 나올 때 함께 설계해요.

👨‍💻 개발자
오프라인 코스에는 해당 기능이 필요 없는 거 맞나요?
🙋 기획자

맞아요. 오프라인은 차수 배정으로 그룹 분리가 되니까 커리큘럼 레벨 대상 조건이 따로 필요 없어요.

코스 유형per-curriculum 대상 분기 필요?
오프라인 단독 (Phase 1)❌ 불필요 — 차수로 충분
온라인 단독❌ 현재 불필요
블렌디드 (Phase 3)✅ 필요해짐
👨‍💻 개발자
차수(class)별 인원도 권한이나 대상에 대한 제한을 둘 수 있나요?
🙋 기획자

수강 등록 방식에 따라 달라요. 그리고 용어 하나 먼저 짚을게요 — 저희 스펙에서 차수=Session, 회차=Class예요.

강제 할당 모드 → 차수별 그룹 배정 가능 ✅
1차수 → [개발팀] [기획팀] 배정
2차수 → [영업팀] [마케팅팀] 배정

그룹 칩 + 할당 한도(limit) 입력으로 정원 자동 계산됩니다.

자율 수강 모드 → 차수별 제한 없음

신청 권한 그룹은 코스 레벨에서만 설정됩니다. 차수별 개별 신청 제한은 없어요.

회차 레벨 → 정원 개별 오버라이드 가능 ✅

같은 차수 안에서도 회차(수업일)마다 정원이 다를 수 있어 회차별 개별 정원 설정이 됩니다 (session.capacity override). 개별 설정된 회차는 바이올렛 테마로 구분돼요.

📌 2026-05-06 추가 결정 — 출석률 & 변경 정책
👨‍💻 개발자
출석 회차를 변경/추가/삭제 하게 될 때 학습자 출석률 재계산 정책을 정해야 해요. 분모가 핵심입니다.
  • Option A — 차수 전체 회차 수 (미래 포함). 회차 추가 즉시 분모 증가 → ELIGIBLE 강등 발생
  • Option B — 마감된 회차 수만. 미래 회차 추가/삭제는 snapshot 영향 없음, re-fanout 불필요
🙋 기획자

Option B로 갑니다. 분모는 마감된 회차 수만 카운트.

왜 B인가

근거설명
학습자 공정성안 열린 회차로 rate 떨어지면 부당. 회차 추가는 보통 학습자 책임이 아닌 사유로 일어남
운영 가능성차수·회차 추가는 일상. 매번 re-fanout 트리거되면 시스템·운영팀이 못 버팀
수료 시점 동일코스 종료 시점에는 A·B 결과가 같음. 차이는 진행 중 표시에서만
법정의무 정의 일치75% 법정 기준은 "최종 출석률". 진행 중 표시는 "현재까지의 출석률"이 자연스러움

비유로 보면

A는 시험 진행 중에 안 푼 문제까지 0점 처리한 점수. B는 지금까지 푼 문제 중에서 몇 점 맞았는지를 보여주는 점수.

10회차 코스, 3회차 마감, 2회 출석한 학습자 기준:

계산표시학습자 인상
A2 ÷ 1020%"망했네…"
B2 ÷ 367%"남은 7회 챙기면 회복 가능"

보완 정책 5가지

#정책
1학습자 앱에 마감된 N회차 중 M회 출석 (X%) 형식으로 분자·분모 모두 노출
2미래 회차 추가 시 학습자에게 푸시·메일 알림 ("일정 1개 추가됨")
3수료/미수료 라벨은 차수 종료 후에만 노출 — 진행 중에는 출석률 숫자만 표시 2026-05-06 수정: UI 부담으로 "현재까지" 별도 라벨 제거
4마감된 회차 삭제는 admin 이중 확인 + audit log + re-fanout. "이 작업으로 N명 출석률이 재계산됩니다" dialog 필수
5회차 추가·삭제 모두 audit log에 [시점·작업자·영향 인원] 기록
👨‍💻 개발자
2차수였던 코스에 1차수가 늘어났고, 회차 10개도 추가됐어요. 학습자는 2차수 등록·완료해서 수료 표시까지 떴는데, 이렇게 되면 사용자 결과는 어떻게 됩니까?
🙋 기획자

단위 합의: 출석률은 차수(Session) 단위

분모 = 학습자가 등록된 차수의 마감된 회차 수
분자 = 그중 출석/지각/조퇴 회차 수

학습자는 한 차수에만 등록되니까 다른 차수에 뭐가 일어나든 본인 출석률·수료에 영향 없습니다.

시나리오 매트릭스

어떤 변화?2차수 등록·수료자에게 결과
새 차수(3차수) 추가영향 없음 수료 유지
새 차수(3차수)에 회차 10개 추가영향 없음 수료 유지
본인 차수(2차수)에 회차 10개 추가 — 이미 종료(마감)시스템에서 차단 수료 유지
본인 차수(2차수)에 회차 10개 추가 — 진행 중분모 증가 임시 수료 풀림 → "진행 중"
질문 시나리오는 첫 두 케이스 → 사용자는 영향 없음. 수료 표시 그대로.

정책 약속 3개

#약속
1차수가 종료(closed)되면 그 차수의 출석률·수료는 immutable. 사후 변경 불가
2종료된 차수에는 회차 추가/삭제가 시스템에서 차단. UI 버튼 비활성화
3진행 중 차수에 회차 추가 시 학습자 알림 + "회사 사정으로 추가됨" 안내 자동 발송
한 줄 요약 — "내 차수에서 내 약속(75% 출석)을 지켰으면 끝."
👨‍💻 개발자
솔직히 코스 오픈하고 나서도 계속 차수·회차를 추가하는 운영 케이스가 있긴 하지 않을까요? 많지는 않겠지만요.
🙋 기획자

솔직히 — 꽤 자주 있습니다. 운영 일상이에요.
